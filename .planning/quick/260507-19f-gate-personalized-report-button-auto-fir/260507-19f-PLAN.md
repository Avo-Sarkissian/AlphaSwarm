<objective>
Eliminate the orchestrator-load footgun on the manual "Personalized Report"
(brief-icon) button and verify the FINAL-round auto-fire path works end-to-end.

Background: Phase 41.4 smoke surfaced that clicking the brief icon in the
toolbar opens AdvisoryModal, which calls `advisoryGenerate(cycleId)` on mount
(modals.jsx:357). That POST hits `/api/advisory/{cycle_id}`, the route loads
the orchestrator model (~17 GB unified memory) via
`model_manager.load_model(orchestrator)` in `_run_advisory_synthesis`
(routes/advisory.py:253), runs synthesis, and writes
`advisory/{cycle_id}_advisory.json`. The user reports clicking it shows
"no rendered output" — the synthesis runs but the modal sometimes 409s
(another task in flight), 503s (no portfolio), or polls past the user's
attention span. Either way the manual button must NEVER trigger
orchestrator inference.

The auto-fire path is ALREADY WIRED:
  - `web/app.py:69` passes `on_complete=lambda cycle_id: _auto_trigger_advisory(app, cycle_id)` to SimulationManager
  - `web/simulation_manager.py:207-211` schedules `_auto_trigger_advisory(app, cycle_id)` via `asyncio.create_task` after the sim task completes successfully
  - `_auto_trigger_advisory` (simulation_manager.py:22-49) uses httpx ASGITransport to POST `/api/advisory/{cycle_id}` in-process — no socket, route guards re-used, all errors swallowed/logged

So Part B is wired in code. This plan turns the manual button into a pure
viewer (Part A) and verifies the auto-fire actually fires + the modal
renders the resulting file on next open (Part B verification).

Purpose: Prevent a 17 GB orchestrator load from being one stray click away
on the M1 Max during sim runs. Keep the user-facing UX of "open the brief
modal after FINAL → see the personalized report" intact.

Output: AdvisoryModal becomes a pure viewer. No POST from the frontend ever.
</objective>
