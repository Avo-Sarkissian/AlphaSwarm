## Spacing Scale

Declared values from `variables.css` (source: existing tokens — not re-specified):

| Token | Value | Usage in Phase 36 |
|-------|-------|-------|
| xs | 4px (`--space-xs`) | Markdown paragraph bottom margin, button icon gap, inline `<code>` padding |
| sm | 8px (`--space-sm`) | Gap between list items in rendered markdown, error/status vertical spacing |
| md | 16px (`--space-md`) | Modal internal padding, header/content/footer gap, markdown h2 top margin |
| lg | 24px (`--space-lg`) | Modal outer padding, table cell padding (matches existing `CyclePicker.vue` convention) |
| xl | 32px (`--space-xl`) | Markdown h1 top margin, section break spacing |

Exceptions:
- **Modal size:** `80vw` width, `80vh` height (per D-06). Not on the 4-point scale because it is a proportional viewport constraint, not a spacing value.
- **Modal max-width:** 1200px (prevents absurdly wide lines on ultra-wide monitors — 80ch equivalent for readable prose).
- **Modal min-width:** 640px (below this, markdown tables become unreadable — fallback to `95vw` on mobile).
- **Button heights:** 32px (close button, action buttons) — matches `CyclePicker.vue` `.cycle-picker-btn` convention.
- **Markdown line-height:** 1.6 for prose in rendered report — slightly looser than `--line-height-body` (1.5) because long-form reading benefits from more vertical breathing room. This is a rendered-content concern, not a UI chrome concern.

## Interaction Contract

### ControlBar — Report Button (D-08)

- "Report" button appears in `ControlBar.vue` ONLY when `snapshot.value.phase === 'complete'`.
- Click emits `@open-report-viewer` with no payload.
- `App.vue` sets `showReportViewer = true`.
- Button is visible alongside the existing Stop button (both in the complete-phase row). Order: Report (left) → Stop (right).

### ReportViewer — Open Flow (D-09)

1. `v-if="showReportViewer"` triggers mount of `ReportViewer.vue`.
2. On `onMounted`, component calls `GET /api/replay/cycles?limit=1` (reuses existing endpoint) to resolve the most recent completed `cycle_id`. This is the same source of truth used by the interview flow.
3. With `cycle_id` in hand, component calls `GET /api/report/{cycle_id}`.
4. **If 200:** parses `{ cycle_id, content, generated_at }`, runs `marked.parse(content)` then `DOMPurify.sanitize(html)`, injects into `.report-viewer__markdown` via `v-html`. Footer shows `Generated <relative time>` and `Regenerate Report` button.
5. **If 404:** shows empty state in content area (`No report generated yet`). Footer shows `No report generated yet.` and `Generate Report` button.
6. **If other error:** shows inline error above footer: `Could not load report. Try again.` Content area remains in loading state.

### ReportViewer — Generate Flow (D-01, D-02, D-04)

1. User clicks `Generate Report` (or `Regenerate Report`) in the footer.
2. Component calls `POST /api/report/{cycle_id}/generate`.
3. **If 202 Accepted:** footer switches to "Generating report..." with pulsing animation; button becomes disabled with `Generating...` label. Component starts 3-second polling interval calling `GET /api/report/{cycle_id}`.
4. **If 409 Conflict (generation already in progress):** assume a generation is running, switch to polling state without re-triggering. No error shown — this is a recoverable sync state.
5. **If other error (500, network):** inline error above footer: `Could not start report generation. Try again.` Button re-enables.
6. On each poll tick, GET returns 200 once the file exists. Component stops polling, parses response, transitions to rendered state. Footer shows `Generated just now` and `Regenerate Report` button.
7. If user closes the modal DURING generation, polling stops (cleaned up in `onUnmounted`). The backend task continues — on next open, user can GET the finished report or see it still generating.

### ReportViewer — Close Behavior

- **X button:** Click sets `showReportViewer = false` in `App.vue`.
- **Backdrop click:** Click on the backdrop (target equals backdrop element, not a child) closes the modal.
- **Escape key:** Global `keydown` listener (attached on `onMounted`, removed on `onUnmounted`) closes the modal when `event.key === 'Escape'`.
- No confirmation required — closing is non-destructive (no unsaved state, no destructive action).

### ReportViewer — Error Handling (Claude's Discretion — resolved)

Inline errors above the footer, never blocking. Auto-dismiss on next successful action.

| Scenario | Error Copy |
|----------|-----------|
| Network failure (initial GET) | `Could not load report. Try again.` |
| Network failure (poll tick) | Silent — retry on next tick (no error shown during polling) |
| HTTP 500 (GET) | `Could not load report. Try again.` |
| HTTP 500 (POST generate) | `Could not start report generation. Try again.` |
| HTTP 409 (POST generate — already running) | Silent — switch to polling state |
| Generation timeout (no file after 10 minutes of polling) | `Report generation is taking longer than expected. The task may still complete — try refreshing in a few minutes.` — polling stops after this cap to prevent infinite polling |

### ReportViewer — Polling Lifecycle (D-04)

- Poll interval: 3000ms (D-04)
- Poll starts immediately after successful `POST /generate` (202 response)
- Poll stops on: successful GET with 200, modal close, component unmount, error cap reached (200 poll iterations = 10 minutes)
- Polling is paused implicitly if user closes the modal (Vue component unmount clears the interval in `onUnmounted`)
- Reopening the modal restarts the entire flow (GET → possibly start polling if file still missing)

- "Generate" is the verb already used in CONTEXT.md (D-01, D-04) and in the backend route name (`/generate`)
- "Generate" matches industry convention for AI/LLM-produced artifacts
- "Regenerate" is the natural companion verb, signaling non-destructive replacement

**Cycle ID display (Claude's Discretion — resolved):** Truncated to first 8 characters in the modal header (`Report — a1b2c3d4`). Full UUID is available in the GET response payload if needed for debugging, but the header prioritizes legibility. This matches the convention in `CyclePicker.vue` (`cycle.cycle_id.slice(0, 8)`).

**Polling debounce (Claude's Discretion — resolved):** Polling is canceled on modal close via `onUnmounted` cleanup (`clearInterval(pollIntervalId)`). Additionally, if `cycle_id` changes (shouldn't happen in this flow but defensively), any in-flight poll is cancelled via an `AbortController`. This prevents memory leaks from stale polling on closed modals, addressing the pattern already established in `useWebSocket` teardown (referenced in CLAUDE.md quick-260416-trw).
