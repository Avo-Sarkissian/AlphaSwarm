# Phase 999.1: Auto-Trigger Advisory Synthesis on phase=complete — Research

**Researched:** 2026-05-04
**Domain:** FastAPI async task lifecycle + React frontend effect hooks + WebSocket phase observation
**Confidence:** HIGH

---

## Summary

When the simulation reaches `SimulationPhase.COMPLETE`, `simulation.py` calls
`await state_store.set_phase(SimulationPhase.COMPLETE)` at line 1221 and then
returns the `SimulationResult` to `SimulationManager._run()`. At that point the
background task is still alive for a few more microseconds before
`_on_task_done` fires and releases the lock. No automatic advisory trigger
follows today — the user must open the Advisory panel manually and wait for the
`AdvisoryModal` to call `advisoryGenerate(cycleId)` on mount.

The safest and most maintainable trigger point is **backend-side**: inside
`SimulationManager._on_task_done` (or a new `_on_simulation_complete` hook),
fire a `POST /api/advisory/{cycle_id}` call against the local FastAPI app via
`httpx.AsyncClient` (already a project dependency). This mirrors how the
frontend calls the endpoint today, re-uses all existing guards (409 if wrong
phase, 503 if no portfolio, 409 if task in flight), and requires no new
infrastructure. A frontend-side `useEffect` watcher is the simpler alternative
and avoids backend coupling, but introduces a race: the effect fires when the
React component re-renders after seeing `phase === 'complete'` in the WebSocket
frame, which can be up to 200 ms after the backend sets COMPLETE, and multiple
browser tabs could fire simultaneous POSTs (the backend 409 guard would absorb
duplicates, but it is messier). The backend approach is preferred.

The cycle_id needed for the POST must be extracted from the completed
`SimulationResult` returned by `run_simulation()` and piped into
`_on_task_done` or stored on `SimulationManager` before the done-callback runs.
Currently `_on_task_done` receives only the `asyncio.Task` and has no reference
to the result.

**Primary recommendation:** Add an optional `on_complete` async callback slot
to `SimulationManager`. After `run_simulation()` returns in `_run()`, store
the `cycle_id` on `self`, then in `_on_task_done` (or via the new hook) fire
`POST /api/advisory/{cycle_id}` using `httpx.AsyncClient(base_url=...)`.
Guard against: no portfolio loaded (503 → swallow silently, advisory was
impossible anyway), concurrent task in flight (409 → swallow, it will eventually
complete), and wrong phase (impossible here but guard anyway).

## Architecture Patterns

### Current Simulation Completion Flow

```
simulation.py:run_simulation()
  └─ [line 1221] state_store.set_phase(SimulationPhase.COMPLETE)
  └─ returns SimulationResult(cycle_id=..., ...)

SimulationManager._run()
  └─ await run_simulation(...)       ← result returned here, currently discarded
  └─ [implicitly] task finishes

SimulationManager._on_task_done(task)  ← sync callback, no await allowed here
  └─ self._is_running = False
  └─ self._lock.release()
  └─ log.info("simulation_completed")
```

**Key finding:** `_on_task_done` is a **synchronous** done-callback. It CANNOT
await. Any async work must be scheduled via `asyncio.create_task()` from
within it, or the auto-trigger must be wired inside `_run()` BEFORE the
function returns (while we still can await). [VERIFIED: simulation_manager.py lines 154–173]

### Where cycle_id Lives

`run_simulation()` returns a `SimulationResult` with `cycle_id: str`.
That result is currently not captured in `_run()`:

```python
# simulation_manager.py line 123-135 (current)
await run_simulation(
    rumor=seed,
    ...
)
# B10: do NOT set COMPLETE here — simulation.py already does
```

The `cycle_id` must be stored before `_run()` exits so `_on_task_done` or
a follow-up task can reference it. [VERIFIED: simulation_manager.py]

### Advisory POST Guard Chain

`POST /api/advisory/{cycle_id}` in `routes/advisory.py` checks in order:
1. `_validate_cycle_id` — regex guard
2. `graph_manager / ollama_client / model_manager` not None → 503
3. `portfolio_snapshot` not None → 503
4. `snap.phase == SimulationPhase.COMPLETE` → 409 if not
5. `report_task` in flight → 409
6. `existing_advisory` in flight → 409

[VERIFIED: web/routes/advisory.py lines 53–141]

If the auto-trigger fires after `set_phase(COMPLETE)` (which it always will,
since `_run()` returns only after `run_simulation()` returns), guard #4 is
satisfied. Guards #5 and #6 are benign 409s if something else fired first.
Guards #2 and #3 can produce 503s that should be logged but not surfaced as
user-visible errors.

### In-Process HTTP with httpx.ASGITransport

the same process is:

```python
# Conceptual — not yet in codebase
import httpx
from alphaswarm.web.app import app as fastapi_app

async def _auto_trigger_advisory(app: FastAPI, cycle_id: str) -> None:
    transport = httpx.ASGITransport(app=app)
    async with httpx.AsyncClient(transport=transport, base_url="http://test") as client:
        resp = await client.post(f"/api/advisory/{cycle_id}")
    if resp.status_code not in (202, 409):
        log.warning("auto_advisory_trigger_unexpected", status=resp.status_code, cycle_id=cycle_id)
```

This approach:
- Never opens a network socket (ASGITransport calls the ASGI app directly)
- Re-uses all existing route guards
- Is already testable via `httpx.ASGITransport` in the test suite pattern

Alternative: call `_run_advisory_synthesis()` directly from `SimulationManager._run()`.
This bypasses the HTTP layer and all its guards (portfolio check, phase check,
concurrent task check). It introduces direct coupling between SimulationManager
and the advisory engine and could run advisory concurrently with the simulation's
own cleanup. **Not recommended.** [ASSUMED — design judgment]

### Frontend Side: No Change Required (Preferred Path)

The `AdvisoryModal` already calls `advisoryGenerate(cycleId)` on mount (kick-and-poll
pattern). If the backend auto-trigger has already fired and the file exists, the
GET poll returns 200 immediately. If the backend trigger fired and synthesis is
still in progress, the GET poll returns 404 until done. The modal's UX is
identical either way.

However, the user still has to **open** the Advisory panel manually today
(`ui.advisoryOpen` in `app_v2.jsx` line 139). The feature description says
"users don't need to manually open the panel and click Analyze." This means
the plan must also decide whether to:
1. Auto-open the Advisory modal when `phase === 'complete'` (frontend change), OR
2. Simply pre-warm the synthesis in the background so opening the panel shows
   results instantly (backend-only, no UX change to panel open)

Option 2 is the minimum viable interpretation and avoids any modal-forcing UX.
Option 1 is more complete but requires a `useEffect` in `app_v2.jsx` that
watches `phase` and calls `setUi(u => ({...u, advisoryOpen: true}))`.

[ASSUMED — the exact UX scope should be confirmed, but Option 2 (background pre-warm)
is the safer default and is the recommendation here]

## Common Pitfalls

### Pitfall 1: `_on_task_done` Is Synchronous — Cannot `await`

**What goes wrong:** Developer adds `await client.post(...)` inside
`_on_task_done`. This raises `RuntimeError: coroutine was never awaited` or
silently does nothing depending on Python version.

**Why it happens:** `asyncio.Task.add_done_callback` receives a plain callable,
not a coroutine. The event loop calls it synchronously.

**How to avoid:** Call `asyncio.create_task(_auto_trigger_advisory(app, cycle_id))`
from inside `_on_task_done`. The task is scheduled on the running event loop
and executes asynchronously after the callback returns.

**Warning signs:** If you see the POST never firing despite the callback
appearing to run — check whether you are `await`ing inside the sync callback.

### Pitfall 2: cycle_id Not Available in `_on_task_done`

**What goes wrong:** `_run()` currently discards the `SimulationResult` return
value. `_on_task_done` has no reference to `cycle_id`.

**How to avoid:** In `_run()`, capture the result:
```python
result = await run_simulation(...)
self._last_cycle_id = result.cycle_id
```
Then read `self._last_cycle_id` in `_on_task_done`.

**Race note:** `_on_task_done` fires on the same event loop iteration as the
task completion. `self._last_cycle_id` is set synchronously before the task
ends — there is no race.

### Pitfall 3: Triggering Before Phase Is Set to COMPLETE

**What goes wrong:** The auto-trigger POST fires while the state_store still
shows `ROUND_3`, producing a 409 "advisories only available after simulation
completes."

**Why it happens:** If the POST is wired BEFORE `state_store.set_phase(COMPLETE)`
in `simulation.py`, the guard fails.

**How to avoid:** Trigger only in `_on_task_done` or after `run_simulation()`
returns in `_run()`. `simulation.py` sets `COMPLETE` (line 1221) before
returning `SimulationResult`. By the time `_run()` captures the result,
the phase is guaranteed COMPLETE. [VERIFIED: simulation.py lines 1220–1230]

### Pitfall 4: Portfolio Not Loaded Produces Noisy 503 Errors

**What goes wrong:** If `portfolio_snapshot` is None (CSV not configured),
the auto-trigger POST returns 503. If this is logged as an error, it creates
misleading alarm every time a simulation completes.

**How to avoid:** Treat 503 `holdings_unavailable` as an expected degraded-mode
condition. Log at WARNING level (not ERROR), do not surface to the user.
The advisory feature is already optional when holdings CSV is not configured.

### Pitfall 5: Double-Trigger from Multiple Browser Tabs

**What goes wrong:** If the frontend also triggers on `phase=complete` (Option 1
from Architecture Patterns), two concurrent browsers cause two POSTs. The
backend 409 absorbs the second, but it creates confusing log noise.

**How to avoid:** Stick to backend-only auto-trigger (Option 2). The frontend
modal continues to use the kick-and-poll pattern it already has. The backend
is the single trigger site.

### Pitfall 6: Auto-Trigger Runs While report_task Is In Flight

**What goes wrong:** User opens Report panel at the same time simulation
completes. The auto-advisory trigger hits the 409 "report_generation_in_progress"
guard and never re-tries.

**How to avoid:** Implement a simple retry loop in `_auto_trigger_advisory`:
wait up to N seconds for the report task to clear, then POST again. Or, accept
this as a known limitation: if the user manually clicks Analyze later, they get
the advisory anyway. Document in plan as a known limitation for v1 of this feature.

```python
# simulation_manager.py — _on_task_done() modification
def _on_task_done(self, task: asyncio.Task[None]) -> None:
    self._is_running = False
    self._task = None
    self._pending_shock = None
    self._lock.release()

    if not task.cancelled() and task.exception() is None:
        # Simulation completed successfully — schedule auto-advisory trigger
        cycle_id = getattr(self, "_last_cycle_id", None)
        if cycle_id is not None and self._app is not None:
            asyncio.create_task(
                _auto_trigger_advisory(self._app, cycle_id),
                name=f"auto_advisory_{cycle_id}",
            )
        log.info("simulation_completed")
    elif task.cancelled():
        log.info("simulation_cancelled")
    else:
        exc = task.exception()
        log.error("simulation_failed", error=str(exc))
```

```python
# New helper function — fire POST /api/advisory/{cycle_id} in-process
import httpx

async def _auto_trigger_advisory(app: FastAPI, cycle_id: str) -> None:
    """Fire POST /api/advisory/{cycle_id} in-process after simulation completes.

    Uses httpx.ASGITransport to call the FastAPI app directly (no socket).
    Swallows 409 (concurrent task) and 503 (no portfolio / services unavailable)
    as expected conditions. Logs unexpected statuses at WARNING.
    """
    transport = httpx.ASGITransport(app=app)
    try:
        async with httpx.AsyncClient(transport=transport, base_url="http://testserver") as client:
            resp = await client.post(f"/api/advisory/{cycle_id}")
        if resp.status_code == 202:
            log.info("auto_advisory_trigger_accepted", cycle_id=cycle_id)
        elif resp.status_code == 409:
            log.info("auto_advisory_trigger_skipped_conflict", cycle_id=cycle_id)
        elif resp.status_code == 503:
            log.warning("auto_advisory_trigger_skipped_unavailable", cycle_id=cycle_id)
        else:
            log.warning(
                "auto_advisory_trigger_unexpected_status",
                cycle_id=cycle_id,
                status=resp.status_code,
            )
    except Exception:
        log.exception("auto_advisory_trigger_failed", cycle_id=cycle_id)
```

### SimulationManager constructor needs `app` reference

The `SimulationManager.__init__` currently receives `app_state`, `brackets`,
`on_start`, and `replay_manager`. It does NOT hold a reference to the FastAPI
`app` object, which is needed for `httpx.ASGITransport(app=app)`.

The `app` reference is available in `web/app.py:lifespan()` where
`sim_manager` is constructed. Two options:

**Option A:** Pass `app: FastAPI` as an optional constructor parameter.
Minimal change; testable with a mock app.

**Option B:** Call `asyncio.get_running_loop().create_task(client.post(...))` 
against `http://localhost:8000` via real TCP. Fragile — assumes port 8000.

**Option A is correct.** [ASSUMED — design judgment, consistent with project pattern of passing explicit dependencies]

## Open Questions (RESOLVED)

1. **Should the Advisory panel auto-open when synthesis completes?**
   - What we know: the feature description says "users don't need to manually open the panel and click Analyze." This could mean only the background pre-warm, or it could mean the panel should appear.
   - What's unclear: whether forcing the modal open is desirable UX (it may interrupt the user's current view).
   - RESOLVED: Implement background pre-warm only (Option 2 from Architecture Patterns). Plan 02 UAT confirms the panel shows synthesized data when opened, without requiring an "Analyze" click. Frontend auto-open is deferred.

2. **Should there be a retry if `report_task` is in flight (409 conflict)?**
   - What we know: if the user clicks "Report" and "Advisory" at the same time the simulation completes, the auto-trigger hits 409.
   - What's unclear: how often this race occurs in practice.
   - RESOLVED: Log the 409 as `auto_advisory_trigger_skipped_conflict` at INFO and do not retry in v1. Advisory data is available whenever the user opens the panel next. UAT row U5 validates this graceful degradation.

3. **Does `SimulationManager` need the FastAPI `app` reference, or should the auto-trigger be triggered via a callback?**
   - What we know: passing `app: FastAPI` as a constructor param is clean; passing a callable `on_complete: Callable[[str], Awaitable[None]] | None` is even looser coupling.
   - RESOLVED: Use the `on_complete` callback pattern — set in `web/app.py:lifespan()` as a closure over `app`. Implemented in Plan 01 Task 3 and wired in Plan 02 Task 1. `SimulationManager` remains framework-agnostic.
