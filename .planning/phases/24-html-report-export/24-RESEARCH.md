# Phase 24: HTML Report Export - Research

**Researched:** 2026-04-09
**Domain:** HTML report generation, SVG charting (pygal), Jinja2 templating
**Confidence:** HIGH

## Summary

Phase 24 adds `--format html` to the existing `alphaswarm report` CLI command. The existing report pipeline (ReACT engine + Jinja2 assembler) already collects all 9 data sections as `ToolObservation` objects. The work is adding a parallel `assemble_html()` method on `ReportAssembler` that renders a single master HTML/Jinja2 template (`report.html.j2`). Charts are generated by pygal 3.1.0 as inline SVG strings (no XML declaration, no JavaScript), embedded directly into the HTML template. The result is a fully self-contained `.html` file under 1MB.

The critical technical insight is that pygal's `render()` with `disable_xml_declaration=True` produces a unicode SVG string suitable for inline HTML embedding. The `js=[]` config is mandatory to eliminate external JavaScript dependencies (pygal defaults to loading tooltips from `kozea.github.io`). The `file://` CSS references (pygal's `style.css` and `graph.css`) are already embedded inline within the SVG by pygal's own resolution logic, so no external files are needed.

**Primary recommendation:** Use pygal 3.1.0 with a custom `Style` subclass matching the TUI's ALPHASWARM_THEME colors. Generate SVG strings via `chart.render(is_unicode=True)` with `disable_xml_declaration=True` and `js=[]`. Pass pre-rendered SVG strings into the Jinja2 HTML template as context variables. The HTML template handles layout (2-column CSS grid) and dark styling.

### Pattern 1: Chart Builder Functions in report.py
**What:** Pure functions that accept observation data and return inline SVG strings. Each chart function creates a pygal chart object, configures it, adds data series, and returns `chart.render(is_unicode=True)`.
**When to use:** Every SVG chart embedded in the HTML report.
**Example:**
```python
import pygal
from pygal.style import Style

# Custom style matching TUI ALPHASWARM_THEME
ALPHASWARM_CHART_STYLE = Style(
    background="#121212",
    plot_background="#1E1E1E",
    foreground="#E0E0E0",
    foreground_strong="#FFFFFF",
    foreground_subtle="#78909C",
    colors=("#66BB6A", "#EF5350", "#78909C"),  # BUY, SELL, HOLD
    font_family="'Segoe UI', 'Helvetica Neue', Arial, sans-serif",
    label_font_size=12,
    title_font_size=14,
    value_font_size=11,
    guide_stroke_color="#333333",
    major_guide_stroke_color="#444444",
)

CHART_CONFIG = {
    "style": ALPHASWARM_CHART_STYLE,
    "disable_xml_declaration": True,
    "js": [],              # CRITICAL: no external JS for self-contained output
    "width": 600,
    "height": 400,
    "show_legend": True,
    "print_values": True,
    "print_values_position": "top",
    "explicit_size": False, # Let SVG scale responsively
}


def render_consensus_bar(consensus_data: dict) -> str:
    """Render consensus summary as HorizontalBar SVG string."""
    chart = pygal.HorizontalBar(**CHART_CONFIG)
    chart.title = "Final Consensus (Round 3)"
    chart.add("BUY", consensus_data["buy_count"])
    chart.add("SELL", consensus_data["sell_count"])
    chart.add("HOLD", consensus_data["hold_count"])
    return chart.render(is_unicode=True)


def render_round_timeline(timeline_data: list[dict]) -> str:
    """Render round-over-round signal changes as Line chart SVG string."""
    chart = pygal.Line(**CHART_CONFIG)
    chart.title = "Signal Distribution by Round"
    chart.x_labels = [f"Round {r['round_num']}" for r in timeline_data]
    chart.add("BUY", [r["buy_count"] for r in timeline_data])
    chart.add("SELL", [r["sell_count"] for r in timeline_data])
    chart.add("HOLD", [r["hold_count"] for r in timeline_data])
    return chart.render(is_unicode=True)
```

### Pattern 2: assemble_html() Method on ReportAssembler
**What:** New method parallel to existing `assemble()`. Receives same `observations` and `market_context_data`. Generates SVG strings from chart builders, then renders `report.html.j2` with all data and SVG strings as template context.
**When to use:** When `--format html` is specified.
**Example:**
```python
def assemble_html(
    self,
    observations: list[ToolObservation],
    cycle_id: str,
    *,
    market_context_data: list[dict] | None = None,
) -> str:
    """Assemble observations into a self-contained HTML report."""
    # Use a SEPARATE Jinja2 Environment with autoescape=True for HTML
    html_env = Environment(
        loader=FileSystemLoader(str(TEMPLATE_DIR)),
        autoescape=True,    # HTML output needs escaping
        trim_blocks=True,
        lstrip_blocks=True,
    )

    obs_by_tool = {obs.tool_name: obs for obs in observations}

    # Generate SVG charts from observation data
    consensus_svg = ""
    if "bracket_summary" in obs_by_tool:
        consensus_svg = render_consensus_bar(obs_by_tool["bracket_summary"].result)

    timeline_svg = ""
    if "round_timeline" in obs_by_tool:
        timeline_svg = render_round_timeline(obs_by_tool["round_timeline"].result)

    # ... more chart renderers ...

    template = html_env.get_template("report.html.j2")
    return template.render(
        cycle_id=cycle_id,
        generated_at=datetime.datetime.now(tz=datetime.timezone.utc).isoformat(),
        consensus_svg=consensus_svg,
        timeline_svg=timeline_svg,
        # ... all data and SVGs ...
        market_context_data=market_context_data or [],
        observations=obs_by_tool,
    )
```

### Pattern 3: Jinja2 Markup Safety for Inline SVG
**What:** SVG strings from pygal must be injected as raw HTML (not escaped) into the template. Jinja2's `autoescape=True` would escape `<svg>` tags. Use `Markup()` or the `|safe` filter.
**When to use:** Every SVG string passed to the HTML template.
**Example:**
```html
{# In report.html.j2 #}
<div class="chart-panel">
    {{ consensus_svg | safe }}
</div>
```
**Important:** Only use `|safe` on SVG strings generated by our own code (pygal output). Never on user-supplied data.

### Pattern 4: CLI Format Branching
**What:** Add `--format` argument to report subparser. Branch in `_handle_report()` after observations are collected.
**When to use:** CLI integration point.

### Anti-Patterns to Avoid
- **External resource references in HTML:** Never link to CDN stylesheets, fonts, or scripts. Everything must be inline for self-contained output.
- **Using `autoescape=False` for HTML template:** The existing markdown assembler uses `autoescape=False`. The HTML template MUST use `autoescape=True` (or a separate Environment) to prevent XSS from agent-generated content. SVG strings use `|safe` filter explicitly.
- **Sharing Jinja2 Environment between markdown and HTML:** Different autoescape settings require separate Environment instances. Create the HTML env in `assemble_html()` or as a class attribute.
- **Using `render_data_uri()` instead of inline SVG:** Data URI approach base64-encodes the SVG, making it ~33% larger and not CSS-styleable from the page. Inline SVG is smaller and inherits page styles.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| SVG chart generation | Manual SVG path/rect elements | pygal chart objects | Axis calculation, label positioning, legend layout, scaling are complex; pygal handles all of it |
| Dark theme color system | Custom CSS variables for charts | pygal `Style` class | Style propagates to axes, labels, guides, legend, tooltips automatically |
| CSS grid layout | Custom flexbox with manual breakpoints | CSS Grid `grid-template-columns` | 2-column responsive layout is a 3-line CSS grid declaration |
| HTML escaping | Manual string sanitization | Jinja2 `autoescape=True` | Agent rationale text may contain angle brackets, quotes; Jinja2 handles all edge cases |

**Key insight:** The entire charting problem is solved by pygal. The entire templating problem is solved by Jinja2 (already in the project). The only new code is: (1) chart builder functions that map data shapes to pygal API calls, (2) the HTML template file, and (3) the `assemble_html()` method that glues them together.

## Common Pitfalls

### Pitfall 1: External JavaScript in pygal SVG Output
**What goes wrong:** pygal defaults to including `<script href="https://kozea.github.io/pygal.js/2.0.x/pygal-tooltips.min.js">` in SVG output. This breaks the self-contained requirement and fails when opened offline.
**Why it happens:** pygal's default `js` config tuple contains an external URL for tooltip interactivity.
**How to avoid:** Set `js=[]` in every chart configuration. This removes all external JS references.
**Warning signs:** HTML file makes network requests when opened; tooltips work in online mode but not offline.

### Pitfall 2: XML Declaration in Inline SVG
**What goes wrong:** pygal's `render()` includes `<?xml version="1.0" encoding="utf-8" ?>` by default. When embedded inside HTML, this causes rendering issues in some browsers.
**Why it happens:** SVG is valid XML; the declaration is standard for standalone SVG files.
**How to avoid:** Set `disable_xml_declaration=True` in chart config. This also makes `render()` return unicode string instead of bytes.
**Warning signs:** Browser shows raw XML text instead of chart; `render()` returns `bytes` instead of `str`.

### Pitfall 3: Jinja2 Escaping SVG Markup
**What goes wrong:** With `autoescape=True`, Jinja2 escapes `<svg>`, `<rect>`, `<path>` etc. into `&lt;svg&gt;`, rendering the chart as visible HTML entities instead of a graphic.
**Why it happens:** Autoescape is correct for HTML safety but SVG strings are trusted output from our own code.
**How to avoid:** Use `{{ svg_string | safe }}` filter in the template for pygal-generated SVG strings ONLY. Never use `|safe` on user-supplied data (agent rationale, rumor text).
**Warning signs:** Chart area shows raw SVG source code as text.

### Pitfall 4: File Size Exceeding 1MB
**What goes wrong:** HTML file exceeds the 1MB success criterion.
**Why it happens:** Each pygal SVG chart is typically 15-50KB. With 9 sections and up to 3 ticker mini-charts, total SVG is ~200-400KB. CSS and HTML structure add ~10KB. Agent data tables with 100 agents could add significant text. Very unlikely to exceed 1MB.
**How to avoid:** Use `pretty_print=False` (default) for compact SVG output. Truncate social post content (already truncated to 80 chars in markdown templates). Monitor total file size in tests.
**Warning signs:** File size grows linearly with agent count in table sections.

### Pitfall 5: Separate Jinja2 Environment for HTML
**What goes wrong:** Modifying the existing `ReportAssembler._env` to use `autoescape=True` breaks all existing markdown template rendering.
**Why it happens:** The existing environment is configured for markdown output (`autoescape=False`).
**How to avoid:** Create a separate `Environment` instance for HTML rendering. Either as a second class attribute or instantiated in `assemble_html()`.
**Warning signs:** Markdown reports start showing `&amp;` or `&lt;` in output after HTML feature is added.

### Pitfall 6: pygal HorizontalBar Data Shape
**What goes wrong:** pygal `HorizontalBar.add()` expects a single value or list, not a dict. Passing the raw `consensus_data` dict directly causes errors.
**Why it happens:** The graph query returns `{"buy_count": 50, "sell_count": 30, ...}` but pygal expects `chart.add("BUY", [50])` or `chart.add("BUY", 50)`.
**How to avoid:** Chart builder functions explicitly map from graph query dict keys to pygal `add()` calls.
**Warning signs:** TypeError or empty charts.

## Open Questions

1. **D-04 `render_inline()` Correction**
   - What we know: pygal has no `render_inline()` method. The correct approach is `render(is_unicode=True)` with `disable_xml_declaration=True`.
   - What's unclear: Nothing -- this is a confirmed correction.
   - Recommendation: Implement using `render(is_unicode=True)` + `disable_xml_declaration=True`. The intent of D-04 (inline SVG, CSS-styleable, smallest size) is fully met by this approach.

2. **Bracket Distribution Chart Data Source**
   - What we know: D-01 says "BUY/SELL/HOLD stacked bars per bracket" which maps to `bracket_narratives` observation data (has buy_count, sell_count, hold_count per bracket).
   - What's unclear: Whether this should be a single stacked chart with all 10 brackets, or 10 individual mini-charts.
   - Recommendation: Single `pygal.HorizontalStackedBar` with 10 bracket rows, 3 stacked series (BUY/SELL/HOLD). More compact and visually coherent than 10 separate charts.

3. **Ticker Consensus Data for Mini-Charts (D-03)**
   - What we know: `read_market_context()` returns per-ticker rows with `majority_signal`, `majority_pct`, `consensus_score`. But D-03 wants per-bracket breakdown for each ticker.
   - What's unclear: Whether per-ticker per-bracket data is available from existing graph queries. `TickerConsensusSummary` nodes may contain bracket-level data.
   - Recommendation: If per-ticker bracket data is not available from existing queries, use the market_context data to show overall ticker signal distribution (majority signal + percentage) as a simpler HorizontalBar per ticker. This still mirrors the TUI TickerConsensusPanel which shows aggregated signals.

## Data Shape Reference

For chart builder functions, the observation data shapes from graph queries are:

### consensus_summary (bracket_summary tool)
```python
{"buy_count": 50, "sell_count": 30, "hold_count": 20, "total": 100}
```

### round_timeline (round_timeline tool)
```python
[
    {"round_num": 1, "buy_count": 45, "sell_count": 35, "hold_count": 20, "total": 100},
    {"round_num": 2, "buy_count": 48, "sell_count": 32, "hold_count": 20, "total": 100},
    {"round_num": 3, "buy_count": 50, "sell_count": 30, "hold_count": 20, "total": 100},
]
```

### bracket_narratives (bracket_narratives tool)
```python
[
    {"bracket": "Quants", "buy_count": 8, "sell_count": 1, "hold_count": 1,
     "avg_confidence": 0.85, "avg_sentiment": 0.72},
    # ... 10 brackets total
]
```

### market_context (passed separately, not from ReACT tools)
```python
[
    {"ticker": "TSLA", "company_name": "Tesla Inc", "last_close": 185.40,
     "majority_signal": "SELL", "majority_pct": 0.72, "consensus_score": 0.68,
     "pe_ratio": 62.1, "price_change_30d_pct": -12.3, ...},
]
```

### key_dissenters, influence_leaders, signal_flip_analysis, entity_impact, social_post_reach
These sections are rendered as HTML tables (not charts) in the HTML report. Their data shapes are visible in the existing `.j2` templates. No chart rendering needed -- just HTML table formatting within the master template.
