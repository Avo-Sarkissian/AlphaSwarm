<objective>
Add `suspend()` and `resume()` methods to `ResourceGovernor` that flip the existing `_resume_event` gate **without** touching the governor state machine. This is the minimal intervention that lets `run_simulation()` pause all agent acquisitions during an inter-round shock authoring window without tripping THROTTLED/PAUSED states.

**Revision note (2026-04-11 — reviews):** Both reviewers flagged the resume() semantics under concurrent memory pressure. Codex rated it HIGH; Gemini rated it LOW. The decision from this revision: **`resume()` MUST only set `_resume_event` if `self._state == GovernorState.RUNNING`**. If the monitor loop moved the governor to PAUSED or CRISIS during the shock window, `resume()` is a no-op — the monitor loop retains authority over the gate. The simulation's outer `try/finally` still calls `resume()` unconditionally on the caller side, but the callee now defers to the state machine when a real memory-pressure event is active.

This resolves the Codex HIGH concern (race where resume() releases agents during real memory pressure) and the Gemini LOW concern (user confusion) by making the invariant enforced at the callee, not just documented at the caller.

Purpose: Deliver the smallest, most surgical primitive first — per CONTEXT.md D-01..D-03 and STATE.md note "Governor suspend/resume must be first deliverable in shock phase — prevents false THROTTLED states."

<threats>
  - ID: T-03
    Description: Calling `suspend()` while governor is in PAUSED or CRISIS state creates a race window where `resume()` can release agents during a real memory pressure event, bypassing the state machine's gate.
    Mitigation (REVISED): The callee `resume()` now guards: `if self._state == GovernorState.RUNNING: self._resume_event.set()`. If the monitor loop moved state to PAUSED/CRISIS while the shock window was open, `resume()` logs and is a no-op — the monitor loop will re-set the event when pressure clears via `_update_resume_event()`. This removes the race entirely; the caller no longer needs to pre-check state.
  - ID: T-04
    Description: Regression of the 7-bug governor deadlock fixed in commits 9bd6dfa / 85cade8 — any code change in governor.py risks breaking the monitor lifecycle.
    Mitigation: Keep the diff to two new methods appended after existing ones; do NOT modify `acquire()`, `_monitor_loop`, `stop_monitoring()`, `_update_resume_event()`, or any other existing function. Run the full test_governor.py suite after edit to confirm no existing tests regress.
  - ID: T-05
    Description: Silent no-op — if `_resume_event` is somehow replaced elsewhere, `suspend()` flips a dead event.
    Mitigation: Tests assert observable behavior (acquire() blocks / unblocks) rather than just method presence; no code path in governor.py reassigns `_resume_event` after __init__.
  - ID: T-05b
    Description: REVIEWS-ADDED — resume() respects RUNNING state, but a newly-cleared event by a subsequent _update_resume_event call during an actual memory-pressure pause could still be accidentally set if suspend/resume races with the monitor loop in an unexpected order.
    Mitigation: Tests prove the guard holds across at least one monitor tick (test_suspend_does_not_bypass_memory_pressure_state). If the guard ever fails, agents would be released during real memory pressure — a hard constraint violation — and the test will catch it before merge.
</threats>

    ```python
    def suspend(self) -> None:
        """Block new acquire() calls by clearing the resume event.

        Used to pause agent acquisitions during an inter-round shock
        authoring window (Phase 26). Per 26-CONTEXT.md D-01..D-03, this
        method deliberately bypasses the governor state machine:

        - Does NOT touch self._state, self._crisis_start, or
          self._consecutive_green_checks
        - Does NOT interact with the monitoring loop (which keeps running)
        - Does NOT call stop_monitoring() (which resets the TokenPool)

        The monitor loop's own authority over _resume_event (via
        _update_resume_event()) is unaffected during shock pause because the
        loop only fires on memory-pressure transitions, and inter-round gaps
        have zero active agents. If pressure does spike during the shock
        window, the monitor loop will re-clear this event on its next tick;
        resume() then defers to that state (see resume() docstring).

        Idempotent. Safe to call multiple times.
        """
        self._resume_event.clear()
        log.info("governor_suspended", reason="shock_window")

    def resume(self) -> None:
        """Unblock new acquire() calls ONLY if governor state is RUNNING.

        Per 2026-04-11 reviews revision (Codex HIGH / Gemini LOW concern):
        the callee now respects the authoritative state machine. If the
        monitor loop transitioned the governor to PAUSED or CRISIS during
        the shock window (e.g. memory pressure spiked while the user was
        typing), this method is a safe no-op — the monitor loop's own
        _update_resume_event() will re-set the gate when pressure clears.

        This prevents a subtle race where resume() could release agents
        during a real memory-pressure event, violating the core memory-
        safety invariant from CLAUDE.md (hard constraint).

        Idempotent. Safe to call under all governor states.

        Mirror of suspend() for the common RUNNING path. Per CONTEXT.md D-02.
        """

            self._resume_event.set()
            log.info("governor_resumed", reason="shock_window")
        else:
            log.warning(
                "governor_resume_deferred_memory_pressure",
                reason="shock_window",
                governor_state=self._state.value,
            )
    ```
