<threat_model>
<threats>
  - ID: T-10
    Description: `_poll_snapshot` re-pushes `ShockInputScreen` on every 200ms tick while shock_window stays set, stacking modals and flooding the queue (Pitfall 2 from RESEARCH).
    Mitigation: Rising-edge latch on `self._shock_window_was_open` instance attribute. Only push on False→True transition; reset latch on True→False (when simulation closes the window after draining the queue).
  - ID: T-11
    Description: Sync callback attempts to call async `submit_shock`, returning a coroutine that never runs (Pitfall 4 from RESEARCH).
    Mitigation: Use `self.run_worker(self.app_state.state_store.submit_shock(text), exclusive=False)` to schedule the coroutine on Textual's worker manager. This is the same pattern Textual docs recommend for sync→async bridging.
  - ID: T-12
    Description: Empty string submitted via Enter (user presses Enter with no input) gets persisted as a shock.
    Mitigation: In `on_input_submitted`, dismiss with `text.strip() or None` — empty becomes None, which the simulation treats as "no shock" (D-07). A dedicated test `test_shock_input_screen_empty_enter_dismisses_with_none` guards this behavior against regression (added per 2026-04-11 reviews revision).
  - ID: T-13
    Description: Esc binding conflicts with existing global Esc on dashboard.
    Mitigation: `ShockInputScreen` is a pushed modal Screen; its BINDINGS are scoped to the screen, not the app. Textual routes Esc to the topmost screen first. Mirrors how `RumorInputScreen` and `InterviewScreen` already handle Esc without conflict.
  - ID: T-14
    Description: ShockInputScreen introduces new color or text-style tokens not in the UI contract, breaking the design system.
    Mitigation: CSS copied verbatim from 26-UI-SPEC.md §Layout and Composition. No new hex codes, no new text-style values beyond `bold` and regular.
</threats>

            Extracted from _poll_snapshot so it can be tested without mounting
            the full dashboard widget tree (per Codex review MEDIUM on Plan 04).
