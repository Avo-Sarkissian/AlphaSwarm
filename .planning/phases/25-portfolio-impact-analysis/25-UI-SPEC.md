Rendering notes:
- Numbers unformatted as integers; percentage with one decimal place (e.g., `72.0%`)

- In markdown, rendered as a single italic line: `*Coverage: 18/25 equity holdings matched to swarm consensus (72.0%)*`

### Table Headers (Matched Positions)

Exact column labels in exact order:

| Column | Header Text |
|--------|-------------|
| 1 | `Ticker` |
| 2 | `Shares` |
| 3 | `Market Value` |
| 4 | `Swarm Signal` |
| 5 | `Agreement` |
| 6 | `Entity` |
| 7 | `Avg Sentiment` |

**Note on "Agreement" vs "Confidence":** Per RESEARCH.md Open Question #2, the value is `majority_count / total_mentions`, which is percent agreement, not per-agent Ollama confidence. The column header MUST read `Agreement` to avoid misleading the user. The underlying data key stays `confidence` per D-11 contract, but the rendered label is `Agreement`.

### Cell Formatting Rules (Both Tables)

| Field | Format | Example |
|-------|--------|---------|
| Ticker | Uppercase as-is from CSV | `AAPL` |
| Shares | 4 decimal places, no thousands separator | `101.3071` |
| Market Value | `$` prefix, 2 decimals, thousands separator | `$26,416.56` |
| Swarm Signal | Uppercase word only | `BUY` |
| Agreement | Integer percent with `%` suffix | `82%` |
| Entity | As returned by `read_entity_impact()` | `Apple Inc` |
| Avg Sentiment | 2 decimals, signed when negative | `0.43`, `-0.12` |

**Why integer percent for Agreement (not 1 decimal like Phase 24 Market Context uses):** Agreement is a discrete ratio over 100 agents; fractional precision is false specificity. Planner rationale: 82% is honest, 82.3% suggests resolution the data does not have.

### Empty States

| Condition | Exact Copy | Rendering |
|-----------|------------|-----------|
| `--portfolio` flag not provided | Section is entirely omitted from report (no placeholder). This is the default, no-regression behavior per D-16. | N/A — no DOM node emitted |
| `--portfolio` provided, path does not exist | CLI prints to stderr: `Portfolio file not found: {path}. Continuing without portfolio section.` Report generates without portfolio section. | stderr only |
| `--portfolio` provided, CSV parsed, zero equity rows | Matched card shows: `No equity holdings found in the provided portfolio file.` Gaps card omitted. | One `<p>` inside `.section`, `color: #78909C`, font-size 14px |
| `--portfolio` provided, all holdings matched (zero gaps) | Gaps card shows: `All equity holdings have swarm coverage in this simulation.` | One `<p>` inside `.section`, `color: #78909C`, font-size 14px |
| `--portfolio` provided, all holdings are gaps (zero matches) | Matched card shows: `No held tickers match entities the swarm analyzed this cycle.` Gaps card renders normally. | One `<p>` inside `.section`, `color: #78909C`, font-size 14px |

### Error States

| Condition | Exact Copy | Rendering |
|-----------|------------|-----------|
| CSV missing expected header row (row 3) | CLI prints to stderr: `Portfolio CSV appears malformed: expected header row with 'Symbol' column at line 3. Continuing without portfolio section.` | stderr only; report generates without section |
| CSV currency parse failure on any row | CLI prints to stderr: `Could not parse market value '{raw_value}' on row for {ticker}. Skipping row.` Row is skipped; parse continues. | stderr only; valid rows still render |
| `read_entity_impact()` raises | Exception bubbles up as it would for any other tool; no special handling. Tool observation absent, section absent. | Existing ReportEngine error path |

## Markdown Template Structure (Prescriptive)

The executor MUST structure `10_portfolio_impact.j2` exactly as follows (conditional blocks included):

```jinja
## Portfolio Impact

*Coverage: {{ data.coverage_summary.covered }}/{{ data.coverage_summary.total_equity_holdings }} equity holdings matched to swarm consensus ({{ "%.1f"|format(data.coverage_summary.coverage_pct) }}%)*

### Matched Positions

{% if data.matched_tickers %}
| Ticker | Shares | Market Value | Swarm Signal | Agreement | Entity | Avg Sentiment |
|--------|--------|--------------|--------------|-----------|--------|---------------|
{% for t in data.matched_tickers -%}
| {{ t.ticker }} | {{ "%.4f"|format(t.shares) }} | ${{ "{:,.2f}".format(t.market_value) }} | {{ t.signal }} | {{ (t.confidence * 100)|round|int }}% | {{ t.entity_name }} | {{ "%.2f"|format(t.avg_sentiment) }} |
{% endfor %}
{% else %}
*No held tickers match entities the swarm analyzed this cycle.*
{% endif %}

### Coverage Gaps

{% if data.gap_tickers %}
The following equity holdings had no corresponding entity in this simulation run:

| Ticker | Shares | Market Value |
|--------|--------|--------------|
{% for g in data.gap_tickers -%}
| {{ g.ticker }} | {{ "%.4f"|format(g.shares) }} | ${{ "{:,.2f}".format(g.market_value) }} |
{% endfor %}
{% else %}
*All equity holdings have swarm coverage in this simulation.*
{% endif %}
```

**Note:** The `"{:,.2f}".format(...)` Python-style format is used because Jinja2's built-in `format` filter requires `%` syntax. Since `ReportAssembler._env` is standard Jinja2, the planner should verify which syntax the existing templates use and align. If `07_entity_impact.j2` uses `"%.2f"|format`, then Phase 25 SHOULD match that style; pass thousands-separator formatting via a custom filter or precompute in the tool function. **Prescribed fallback:** precompute `market_value_display` as a pre-formatted string in `_portfolio_impact_tool` — this keeps the template pure and avoids Jinja2 formatting quirks.

## Anti-Patterns (Phase 25 Specific)

The ui-checker and ui-auditor should flag any of the following:

1. **New hex colors** — any color value not in the table above
2. **New font sizes** — anything outside {14, 16, 18, 24} px
3. **New font weights** — anything outside {400, 600}
4. **Icons** — no SVG icons, no emoji in cell content, no dingbats
5. **Italics for emphasis** — reserved only for the meta coverage summary line and empty-state copy
6. **Color-coded coverage percentage** — the number is never red/green
7. **Sorting indicators / interactive headers** — no `<th>` with cursor or arrows
8. **Embedded charts inside portfolio section** — no SVG charts (matched tickers already appear in the Phase 24 ticker-grid at the bottom of the HTML report if the planner wires `market_context_data`; do not duplicate here)
9. **Subsection `<h3>` inside a single card** — use two separate `.section` cards instead
10. **Numeric precision mismatch** — shares always 4 decimals, market value always 2 decimals, agreement always integer %, sentiment always 2 decimals with sign
