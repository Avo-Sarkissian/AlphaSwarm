### Tier 1 ‚Äî Data plane corruption (BLOCKING ‚Äî silently corrupts the v6.0 advisory deliverable)

**D-01: Cypher signal/round normalization (LOCKED)**
- Rewrite the 5 read methods on `GraphStateManager` to compare against lowercase signal values that match `SignalType` enum (`types.py:50-56`):
  - `_read_consensus_summary_tx` (`graph.py:1118`) ‚Äî `'BUY'` ‚Üí `'buy'`, `'SELL'` ‚Üí `'sell'`, `'HOLD'` ‚Üí `'hold'`
  - `_read_round_timeline_tx` (`graph.py:1155`) ‚Äî same three replacements
  - `_read_bracket_narratives_tx` (`graph.py:1194`) ‚Äî same three replacements
  - `_read_key_dissenters_tx` (`graph.py:1234`) ‚Äî same three replacements + the `bracket_majority` CASE branches (`'BUY'`, `'SELL'`, `'HOLD'`)
  - `_read_entity_impact_tx` (`graph.py:1388`) ‚Äî same three replacements
- Fix `_read_signal_flips_tx` (`graph.py:1342`):
  - `WHERE re.flip_type <> 'NONE'` ‚Üí `WHERE re.flip_type <> 'none'`
  - `re.round_num AS round_num` ‚Üí `re.round AS round_num` (writer stores property as `round` per `graph.py:736`; the alias keeps the return shape stable for callers)
  - `ORDER BY re.round_num` ‚Üí `ORDER BY re.round`
- Add a regression test in `tests/test_graph_reads.py` (or extend the nearest existing report-read test file) that:
  - Writes one Decision per signal value (buy/sell/hold) at round 3 plus one RationaleEpisode with `flip_type='buy_to_sell'`
  - Asserts every read method returns non-zero counts/rows
  - This regression test is what the prior coverage was missing ‚Äî without it, the bug is invisible

### Tier 1 ‚Äî ReACT tool-name unification (BLOCKING ‚Äî prevents report generation)

**D-02: Pick prompt-side names, update registries + assembler (LOCKED)**
- The prompt (`report.py:32-39`) is the user-facing contract ‚Äî the model sees it. Keep prompt names `consensus_summary`, `round_timeline`, `bracket_narratives`, `key_dissenters`, `influence_leaders`, `signal_flips`, `entity_impact`, `social_post_reach`. These are the canonical names.
- Update `cli.py:724-733` registry: rename keys `bracket_summary` ‚Üí `consensus_summary`, `signal_flip_analysis` ‚Üí `signal_flips`. Other six already match.
- Update `web/routes/report.py:314-339` registry: same two renames.
- Update `report.py:218-243` `TOOL_TO_TEMPLATE` and `SECTION_ORDER`: same two renames. Template filenames stay (`01_consensus_summary.j2`, `06_signal_flip_analysis.j2`) ‚Äî only the dict keys change.
- Add an invariant test in `tests/invariants/` that asserts every tool name advertised in `REACT_SYSTEM_PROMPT` exists as a key in both registries and in `TOOL_TO_TEMPLATE` ‚Äî prevents recurrence.

### Tier 1 ‚Äî Narrative `system=` repair (BLOCKING ‚Äî interview context degraded)

**D-03: Add `system` parameter to `OllamaClient.generate()` (LOCKED)**
- `simulation.py:1316` calls `client.generate(model=..., prompt=..., system=...)`; current `generate()` signature (`ollama_client.py:112`) accepts only `model`, `prompt`, `format`, `keep_alive`, `options`. Every call raises `TypeError`, swallowed by the bare `except Exception` at `simulation.py:1325`.
- Resolution: extend `generate()` signature to accept `system: str | None = None` and forward to the underlying `ollama` client (which supports it). Mirrors the symmetry with `chat()` which already supports system messages.
- Why not change the caller: keeping the high-level "system + prompt" call site is more readable for narrative generation than concatenating into one prompt.
- Add a unit test that calls `generate(system=...)` with a fake transport and asserts the kwarg is forwarded.
- Add a regression test that runs `_generate_decision_narratives()` with a fake client and asserts at least one narrative is produced for a single agent (currently produces zero because every call throws).

### Tier 2 ‚Äî UI contract mismatches (BLOCKING for end-user value of Phase 41.1)

**D-04: Replace ReportModalV2 mock with real ReportModal (LOCKED)**
- `frontend/src/components/app_v2.jsx:462` mounts `<ReportModalV2 />` (hard-coded fixture data ‚Äî `frontend/src/components/v2.jsx:220-456`).
- The real `<ReportModal />` already exists in `frontend/src/components/modals.jsx:155-244` and consumes the backend `/api/report/{cycle_id}` flow.
- Swap the mount: replace `<ReportModalV2 onClose={...} />` with `<ReportModal onClose={...} />`.
- `app_v2.jsx:466-470` already handles `historyOpen ‚Üí reportOpen` transition; pass the selected cycle ID through. Currently `onOpenReport` (line 468) sets `reportOpen: true` but no cycle is propagated. Wire `cycleId` state alongside `reportOpen` so `<ReportModal cycleId={ui.reportCycleId} />` receives the historical cycle. Same wiring applies to `<AdvisoryModal />`.
- Delete (or leave but unmount) the `ReportModalV2` export. Do not import it anywhere after this swap.

**D-05: Render `portfolio_outlook` + `items[]` in AdvisoryModal (LOCKED)**
- Backend writes `AdvisoryReport` JSON (`advisory/types.py:28-39`) with keys: `cycle_id`, `generated_at`, `portfolio_outlook` (markdown string), `items[]` (each: `ticker`, `consensus_signal`, `confidence`, `rationale_summary`, `position_exposure`), `total_holdings`, `affected_holdings`.
- Current modal (`modals.jsx:386-394`) only checks `data.content` / `data.narrative`. Both are absent ‚Äî the holdings sidebar is the only thing that ever paints.
- Replace the body so it renders, in order:
  1. Generated-at timestamp + counts (`{affected_holdings} of {total_holdings} holdings impacted`)
  2. `portfolio_outlook` rendered through the existing `renderMarkdown()` helper
  3. A table or list of `items[]` showing ticker, consensus_signal (color-coded BUY/SELL/HOLD), confidence (formatted as %), `rationale_summary`, `position_exposure` (currency-formatted)
- Keep the holdings sidebar untouched.
- Match the visual treatment used by the Claude Design UI ported assets (mono labels, JetBrains Mono, signal color tokens already defined in CSS). Do not invent new design tokens.

**D-06: Historical-cycle generation flow (LOCKED ‚Äî regeneratable, with per-cycle lock)**
- Both POST routes currently 409 unless `state_store` phase is `COMPLETE` (`web/routes/report.py:181`, `advisory.py:91`). This blocks regeneration against persisted cycles after a server restart or when the in-memory machine moved on.
- Resolution: relax the phase guard so generation is allowed for any cycle that exists in Neo4j (i.e., for any `cycle_id` returned by `/api/replay/cycles`). The orchestrator-model contention concern still matters ‚Äî keep the existing `report_task` / `advisory_task` in-flight guards, AND add a per-cycle lock so the same cycle cannot be regenerated concurrently.
- Concretely:
  - Drop the `phase != COMPLETE` checks (lines 181-191 in report.py, 91-100 in advisory.py).
  - Add a Cypher `EXISTS` check: if `(:Cycle {cycle_id: $cycle_id})` does not exist, return 404 with `{"error": "cycle_not_found"}` so the modal can surface a real error instead of polling forever.
  - Keep `report_task`/`advisory_task` global locks (orchestrator-model serialization).
- Frontend: `usePolling.ts` (`frontend/src/hooks/usePolling.ts:54`) currently silently stops at `maxAttempts` with no error surfaced. Add a `status: 'polling' | 'ok' | 'timeout' | 'error'` discriminator OR set `error = new Error('polling timed out after N attempts')` when `n >= maxAttempts`. Modals (`modals.jsx:225-232`, `:418-425`) already render `hadError` ‚Äî surfacing timeout there is a 2-line change once the hook reports it.
- Rationale: the cycle-history UI is decorative if historical cycles can't be regenerated. Per-cycle data is durable (it's in Neo4j); only the in-memory phase machine is ephemeral. Phase machine should not gate access to durable data.

### Tier 3 ‚Äî Test gate cleanup (developer productivity)

**D-07: Remove or fix stale red tests (LOCKED)**
- `tests/test_replay_red.py` is an explicitly-red TDD scaffold from Phase 28 that was never moved to green. Delete the file. The replay route (`web/routes/replay.py:60`) is already implemented; whatever red specs were intended are obsolete.
- `tests/test_report.py:338` expects an `assemble_html()` export from `report.py` that does not exist. HTML export is not a milestone deliverable. Delete only the failing tests that depend on the missing `assemble_html` (preserve the rest of the file). If a `tests/test_report.py` test references the markdown assembler instead, leave that one alone.

**D-08: Make integration tests opt-in (LOCKED)**
- `tests/integration/conftest.py:13-20` auto-applies `@pytest.mark.enable_socket` to every test under `tests/integration/`. With `pytest-socket` installed, default `pytest -q` enables real network for integration subtree, which causes 29-test failures in offline CI/dev (live RSS / yfinance hits).
- Resolution: add a custom marker `integration` registered in `pyproject.toml` under `[tool.pytest.ini_options]` markers, AND change `pytest -q` default to `-m "not integration"`. Then to run integration: `pytest -m integration` (which still gets `enable_socket` via the existing conftest hook).
- Alternative considered and rejected: deleting `enable_socket` injection ‚Äî that breaks integration tests that need network when explicitly opted in.

**D-09: Harden import-linter test (LOCKED)**
- `tests/invariants/test_importlinter_contract.py:27` shells out `uv run lint-imports` without a controlled cache dir. Fails in restricted environments.
- Resolution: pass `env={**os.environ, "UV_CACHE_DIR": str(tmp_path)}` to the subprocess call so the test always uses a writable scratch directory.

### Tier 4 ‚Äî Low-priority hardening (single quick task)

**D-10: ReACT JSON parse hardening (LOCKED)**
- `report.py:192` calls `json.loads(input_json or "{}")` with no recovery. A malformed `INPUT:` line crashes the whole report run.
- Resolution: wrap in try/except `json.JSONDecodeError`. On error, append `OBSERVATION: ERROR: invalid JSON in INPUT line: {error}` to messages and `continue` the loop. Same recovery pattern as the existing "Unknown tool" branch (line 188-190).

**D-11: Reconcile model defaults (LOCKED)**
- `config.py:31` defaults: `qwen3.5:32b` / `qwen3.5:7b`.
- `README.md:169` says `35b` / `9b`.
- Modelfiles (`modelfiles/Modelfile.orchestrator`, `Modelfile.worker`) likely have stale comments.
- Resolution: source of truth is `config.py`. Update README and modelfile comments to match `qwen3.5:32b` / `qwen3.5:7b`. If the user has actually moved to `35b/9b`, update `config.py` instead ‚Äî but per `STATE.md` no recent decision logs that move. Default to keeping `config.py` and updating docs.
