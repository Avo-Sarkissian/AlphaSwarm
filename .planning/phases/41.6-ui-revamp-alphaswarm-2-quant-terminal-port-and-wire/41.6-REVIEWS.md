## Gemini Review

The implementation plans for **Phase 41.6 (UI Revamp — AlphaSwarm-2 Port)** are comprehensive, high-quality, and demonstrate a deep understanding of the technical challenges associated with porting standalone React-CDN assets into a modern ES-module pipeline. The strategy of "waves" allows for gradual validation of the core dashboard before tackling the more complex data-bridging required for AdvisoryV2 and the rich ReportModal. The plans effectively balance the user's "verbatim design" requirement with the engineering need for live-data wiring, using a pragmatic **derive-or-stub** approach to reconcile schema drift.

### Strengths
*   **Mechanical Discipline:** The per-file "transform recipe" for converting CDN-globals to ES modules (including specific mitigations for the `v2.jsx` hoisting trap) is excellent.
*   **Proactive Bug-Fixing:** Integrating the `usePolling` timeout patch (D-19) solves a long-standing silent failure issue that would have undermined the UAT of the new modals.
*   **Security Gates:** The use of automated production-bundle grep gates to prevent mock leakage (KR-41.1-02) and design-literal "filler" leakage (Pitfall #3) is a top-tier engineering practice.
*   **Architectural Alignment:** The decision to place `useWebSocket` above the onboarding gate (Pitfall #6) ensures connection stability across view transitions, and the 6-context tree lift preserves the established data distribution pattern.
*   **Documented Parity:** The creation of a 30-row **Parity Matrix** with explicit KR references ensures that human UAT is objective and grounded in the phase's sanctioned gaps.

### Concerns
*   **Report Markdown Parsing (Medium Severity):** D-20/KR-41.6-12 involves parsing free-form markdown to populate rich UI sections (ConvergenceFlow, Moments, etc.). While the "derive-or-stub" policy mitigates failure, this logic is inherently brittle and might require frequent adjustments as the backend orchestrator's writing style evolves.
*   **`v2.jsx` Maintainability (Low Severity):** Porting `v2.jsx` as a single 801-LOC file (Wave 3 Task 2) preserves design integrity but results in a "mega-component" file that will be difficult to maintain or type-check in the long term.
*   **Agent ID False Positives (Low Severity):** The regex-based citation count in `AdvisoryV2` (KR-41.6-10) may over-count if strings matching the prefix pattern (e.g., "M-12") appear in rationale summaries.

### Suggestions
*   **Parser Centralization:** Move the `ReportModal` section-extraction helpers (parseConvergence, parseMoments) from inline functions into `lib/advisoryParse.ts`. This keeps the components clean and allows the parsers to be tested in isolation later.
*   **Strict Agent ID Matching:** Instead of a generic regex for `agents` derivation, consider matching against the actual agent ID list retrieved from `useAgents()` to eliminate false positives in the `rationale_summary` counts.
*   **CSS "Safety First":** During the Wave 2 Task 4 CSS merge, prioritize AlphaSwarm-2's `:root` tokens. If any archive tokens are needed, rename them (e.g., `--legacy-error`) to avoid clobbering the new design system.

### Risk Assessment: Low-Medium
The overall risk is **Low-Medium**. The mechanical aspects of the port are well-mapped and mitigated by TypeScript and the production grep gates. The "Medium" component of the risk lies in the **manual nature of the CSS reconciliation** and the **heuristic-heavy parsing of markdown** for the rich report views. However, the plan's explicit fallback to `'—'` placeholders for unparseable data ensures that the UI remains functional and "honest" about backend data availability. Proceeding with this plan is highly recommended.

## Codex Review

### Summary

The plans are unusually thorough and generally well-sequenced. The wave split is sensible: foundation, dashboard, modal/advisory surfaces, then net-new surfaces plus UAT. The biggest risks are not conceptual; they are execution hazards: TypeScript syntax inside `.jsx` files, mock data production-bundle leakage, AdvisoryV2/report polling trigger ambiguity, and W3/W4 scope size. If those are corrected before execution, the plan should achieve the phase goal.

### Strengths

- Clear wave ordering with explicit dependency handoffs between W1-W4.
- Strong traceability to decisions D-01 through D-23 and KR registers.
- Good security posture around DOMPurify + marked, localStorage limits, and production mock-grep gates.
- AdvisoryV2 derive-or-stub approach is pragmatic and correctly avoids inventing backend fields.
- KR-41.1 audit is evidence-based and avoids pretending backend gaps are solved in a UI phase.
- Human UAT is built into the plan, which is appropriate for a visual parity-heavy frontend replacement.

### Concerns

- **HIGH: `.jsx` files contain TypeScript syntax in later plans.**
  Plans 03 and 04 use `import type`, generic hooks, typed props, `Record<>`, `as const`, and `boolean | null` inside files still named `.jsx`. That will not parse. Either convert these files to `.tsx` or keep them pure JavaScript.

- **HIGH: mock production-leak strategy is internally inconsistent.**
  Plan 03 top-level imports `SIGNAL_WIRE_SEED` from `../mocks/wire`, then requires zero mock identifiers in production bundles. That will likely fail. The plan should define a real production behavior for mock-backed surfaces: DEV-only dynamic imports, production placeholder, or explicit feature gate.

- **HIGH: AdvisoryV2 may poll without triggering synthesis.**
  The plan wires `advisoryFetch()` through `usePolling`, but the backend requirement says `POST /api/advisory/{cycle_id}` triggers synthesis. If auto-trigger is not guaranteed, AdvisoryV2 will show "No advisory yet" instead of generating one.

- **HIGH: ReportModal rich lift is large scope.**
  D-20 adds markdown parsing for convergence, moments, dissent, influence, and followups. This is likely the riskiest single scope expansion and could produce brittle UI unless fallback behavior is very simple.

- **MEDIUM: CSS reconciliation is underspecified.**
  "Diff first 200 lines" can miss required old classes. Use a class-name inventory from rendered components against both stylesheets instead.

- **MEDIUM: package-lock and dependency reproducibility are missing from W1.**
  `package-lock.json` is not listed, and `npm install` may resolve new versions or require network. Prefer copying or regenerating a lock intentionally and documenting it.

- **MEDIUM: edge direction semantics need locking.**
  InterviewV2 and BracketDeepDive both depend on `source_id`/`target_id` meaning. The plan itself notes uncertainty. This should be verified once and encoded in `edges.ts`.

- **MEDIUM: localStorage validation should be stricter.**
  The grep only prevents backend key names in `localStorage.setItem(...)` calls. It will not catch those keys nested inside a persisted JSON object.

- **LOW: onboarding dev bypass mutates localStorage during render.**
  Better to derive the initial state from `import.meta.env.VITE_SKIP_ONBOARDING` or use an effect.

### Suggestions

- Decide file extensions before execution:
  - Use `.tsx` for `v2`, `interview_v2`, `onboarding`, `bracket_deep`, `settings`, and `history` if typed code is desired.
  - Or strip all TS syntax from `.jsx`.

- Replace top-level mock imports with a deterministic gate:
  - `if (import.meta.env.DEV) { const mod = await import(...) }`
  - In production, render "Signal Wire unavailable in production build" or omit the surface with a KR-linked note.

- Update `api/advisory.ts` to expose both trigger and fetch, with a typed `AdvisoryReport`.
  AdvisoryV2 should trigger generation when appropriate, then poll.

- Keep ReportModal rich parsing conservative:
  - Render canonical markdown first.
  - Only render rich subsections when explicit headings exist.
  - Never infer complex chart data from loose prose.

- Add a preflight task before W1:
  - Ensure `frontend-react-archive/` does not already exist.
  - Capture current git status.
  - Decide package-lock handling.

- Replace CSS diffing with:
  - `rg "className=" frontend/src/components`
  - compare class inventory against `styles.css`
  - only add missing required classes.

- Add a small `normalizeSettingsUiState()` function so localStorage only accepts known UI keys and enum values.

- Add a post-UAT auto/doc task in W4 to update `VALIDATION.md`, summaries, and sign-off artifacts. The current checkpoint says no files modified, but the UAT instructions require document edits.

### Risk Assessment

**Overall risk: HIGH.**

The architecture and sequencing are sound, but the execution plan has several build-breaking and release-blocking risks: TS syntax in `.jsx`, production mock leakage, uncertain advisory trigger behavior, and large W3 scope. These are fixable before implementation. With those corrections, risk drops to **MEDIUM**, mostly due to visual parity/UAT breadth and the intentionally stubbed backend gaps.
