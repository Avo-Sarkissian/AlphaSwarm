<objective>
Fix NR-2: ~80 of 100 force-graph nodes render as grey/HOLD-default after FINAL. The user-stated cause was "adapter reads `agent.signal` but post-FINAL snapshot uses `agent.final_signal`". Code investigation reveals a deeper companion bug: `normaliseSignal` in `frontend/src/adapter/frame.ts` returns `'hold'` for ANY non-buy/sell/hold input — including `null` and `'parse_error'`. The PARSE_ERROR case is real (`SignalType.PARSE_ERROR = "parse_error"` in `src/alphaswarm/types.py`), so the live snapshot legitimately carries `parse_error` strings the adapter is silently relabeling as HOLD. Both fixes ship together: `final_signal ?? signal` preference (per NR-2 directive) AND `parse_error` propagation (deeper root cause discovered during planning).

Purpose: Without distinguishing PARSE_ERROR from HOLD, the user cannot visually quantify how many agents failed model output parsing vs. how many genuinely voted HOLD — the two states are semantically opposite (failure vs. confidence) but render identically. Plan 05's UAT row 7 will go from ❌ → ✅ once both fixes land.

Output: Adapter that propagates the actual signal type; viz that renders parse_error nodes with a desaturated-grey + X marker (distinct from HOLD's yellow); type system that admits the parse_error variant.
</objective>
