# Phase 41.1 Plan 07: Propagate parse_error signal through adapter and viz Summary

One-liner: Adapter now propagates SignalType.PARSE_ERROR ("parse_error") instead of silently relabeling it (and null) as HOLD; viz renders parse_error nodes with desaturated grey + X marker, distinct from HOLD's yellow — closing NR-2.

## What Changed

### Adapter (`frontend/src/adapter/frame.ts`)

**`normaliseSignal` — 6-line semantic diff:**

Before:
```ts
function normaliseSignal(raw: unknown): AgentSignal {
  if (typeof raw !== 'string') return 'hold';      // null → HOLD (NR-2 root cause)
  const s = raw.toLowerCase();
  if (s === 'buy' || s === 'sell' || s === 'hold') return s;
  return 'hold';                                    // 'parse_error' → HOLD (NR-2 root cause)
}
```

After:
```ts
function normaliseSignal(raw: unknown): AgentSignal {
  if (typeof raw !== 'string') return 'parse_error';
  const s = raw.toLowerCase();
  if (s === 'buy' || s === 'sell' || s === 'hold') return s;
  if (s === 'parse_error') return 'parse_error';
  return 'parse_error';
}
```

The single-character change at line 27 ("hold" → "parse_error") + the new `if (s === 'parse_error')` pass-through closes NR-2 at the data-flow level. Lowercase coercion (`.toLowerCase()`) preserved — defensive against any future ALL-CAPS regression in backend SignalType.value.

**`agents.map` — defensive `final_signal ?? signal` read:**

```ts
signal: normaliseSignal(
  (s as Record<string, unknown>).final_signal ?? s.signal,
),
```

Forward-compat with HUMAN-UAT NR-2 directive: if backend ever emits `final_signal` for dissent-resolved agents (Phase 35.1 territory), the adapter prefers it; otherwise the `??` falls through to today's `signal`. Cast through `Record<string, unknown>` because `final_signal` is not declared on the live AgentState shape, but the read must succeed without a TS error.

**Dev console diagnostic:**

```ts
if (import.meta.env.DEV && phase === 'complete') {
  const parseErrors = agents.filter((a) => a.signal === 'parse_error').length;
  if (parseErrors > 0) {
    console.debug(`[adapter] FINAL frame parse_error agents: ${parseErrors}/${agents.length}`);
  }
}
```

Fires once per FINAL frame in dev only — production builds tree-shake the entire branch via Vite's `import.meta.env.DEV` constant folding. UAT can quantify model-output failures from the browser console without DOM inspection.

### Type system (`frontend/src/types.ts`)

`AgentSignal` widened from `'buy' | 'sell' | 'hold'` to `'buy' | 'sell' | 'hold' | 'parse_error'`. `tsc -b --noEmit` exits 0 — no panel narrows the union locally, so the widening required no downstream edits in BracketList/RationaleFeed/KpiStrip/ConsensusRing/etc.

Modals.jsx already used `agent.signal ?? 'parse_error'` as a fallback for the `signal.toUpperCase()` call (line 51), so the widened union retroactively legitimates that defensive read.

### Viz (`frontend/src/components/viz.jsx`)

**Color resolution (`sigColor`):**

```js
const sigColor = (s) => {
  if (s === 'buy') return 'var(--buy)';
  if (s === 'sell') return 'var(--sell)';
  if (s === 'hold') return 'var(--hold)';
  return 'var(--text-3)';                       // parse_error or any unknown
};
```

Was a 3-branch ternary that fell through to `var(--hold)` for non-buy/sell — that fallback was a co-conspirator with the adapter bug. Now explicit 4-branch lookup.

**X marker overlay:**

For both grid and force/radial layouts, parse_error nodes get an `<text>` element parented to the same `<g>` as the circle/rect:

```jsx
{isParseError && (
  <text
    className="parse-error-marker"
    textAnchor="middle"
    dominantBaseline="central"
    fontSize={Math.max(8, r * 1.4)}
    fontWeight={700}
    fill="var(--text-2)"
    pointerEvents="none"
  >×</text>
)}
```

Marker translates with node (parented to same `<g>`), `pointerEvents="none"` lets hover events pass through to the underlying circle. Grid layout uses cell-size-relative font, force/radial uses radius-relative font — both readable at the deterministic-SVG scale established in 41.1-02.

### Tooltip + Legend (`frontend/src/components/app_v2.jsx`)

Tooltip's signal label:
```jsx
{agent.signal === 'parse_error' ? 'PARSE ERROR' : (agent.signal || 'hold').toUpperCase()}
```
Two-word "PARSE ERROR" rather than "PARSE_ERROR" so it reads naturally in the t-row (matches BUY/SELL/HOLD typography).

Legend gains a 4th row before the `legend-divider`: grey chip + ×, label "PARSE ERROR". Inline styles match the in-canvas X marker treatment so the legend is a faithful sample of what's on the graph.

### CSS (`frontend/src/styles.css`)

```css
.sig-parse_error { color: var(--text-2); background: color-mix(in srgb, var(--text-3) 22%, transparent); }
```

Same `color-mix` pattern as `.sig-buy`/`.sig-sell`/`.sig-hold` so the Tooltip's `<strong className="sig-parse_error">` renders with the same chip-style padding/translucent fill — but using `var(--text-3)` (already in palette) rather than introducing a 5th theme token.
