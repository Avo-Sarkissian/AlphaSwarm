## ApiError Error-Banner UX (verbatim)

```jsx
{error && (
  <div className="topbar-error" role="alert" style={{
    padding: '6px 12px',
    background: 'var(--sell)',
    color: 'var(--text)',
    fontSize: 12,
    fontFamily: 'JetBrains Mono, monospace',
    display: 'flex',
    alignItems: 'center',
    gap: 8,
  }}>
    <span>{error}</span>
    <button
      className="btn ghost"
      style={{ marginLeft: 'auto', height: 22, padding: '0 8px' }}
      onClick={() => setError(null)}
    >
      Dismiss
    </button>
  </div>
)}
```

`error` is formatted by `apiErrorMessage(e, verb)` which branches on `e instanceof ApiError` and safely reads `body.detail` (string) or `body.detail.message` (shape used by FastAPI structured error responses). Full `e.body` is never rendered (T-41.1-03-03 mitigation).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] `simStart()` missing required `{seed}` body**
- **Found during:** Task 3 planning (reading `api/simulation.ts`)
- **Issue:** Plan 02 shipped `simStart = () => apiPost(...)` with no argument. FastAPI `POST /api/simulate/start` requires `{seed: string}` body (see `src/alphaswarm/web/routes/simulation.py`). A TopBar "Start" click would have 422'd on every run.
- **Fix:** Changed signature to `simStart(seed: string)` and included `{seed}` body. Return types tightened to the actual backend response schemas (`SimStartResponse`, `SimStopResponse`, `SimShockResponse`).
- **Files modified:** `frontend/src/api/simulation.ts`
- **Verification:** `npm run check` + `npm run build` green; TopBar callsite `await simStart(seed.trim())` type-checks.
- **Committed in:** `1b03b7f` (Task 3 commit)

**2. [Rule 3 — Blocking] Plan 02 adapter dropped backend fields required by Plan 03 contract**
- **Found during:** Task 1 planning (reading `adapter/frame.ts`)
- **Issue:** Backend `AgentState` emits `signal: SignalType | None` + `confidence`; `BracketSummary` emits `buy_count/sell_count/hold_count/total/display_name/avg_confidence`. Plan 02's adapter dropped **all** those fields — storing only `AgentView.confidence` and a derived `BracketSummaryView.consensusSignal` that actually read a non-existent `consensus_signal` backend field. Without these, Plan 03's required contract ("bracket-derived colors on nodes", "buy/sell/hold counts on bracket rows", "avgConfidence %") is unsatisfiable — there is literally no data to derive from.
- **Fix:** Extended `types.ts` (additive only) so `AgentView` now has `signal: 'buy' | 'sell' | 'hold'` and `bracketDisplay: string`; `BracketSummaryView` now has `display / buy / sell / hold / total / avgConfidence` (legacy `consensusSignal` + `agentCount` kept for any downstream reader). Extended `adapter/frame.ts` to pass through the real backend fields (`signal` lowercased via `normaliseSignal`; bracket counts + display_name read directly; `avg_sentiment` filling the legacy `consensusSignal` scalar).
- **Files modified:** `frontend/src/types.ts`, `frontend/src/adapter/frame.ts`
- **Verification:** `npm run check` + `npm run build` green; viz now colors nodes by `a.signal`; panels' `BracketList` renders buy/sell/hold bars.
- **Committed in:** `8ae4c7d` (Task 1 commit)

**3. [Rule 3 — Blocking] Viz early-return violated Rules of Hooks**
- **Found during:** Task 1 self-review (first draft placed the empty-state `return` above the `useMemo` calls)
- **Issue:** Would have shifted the hook-call order between "no agents" and "has agents" renders, triggering React warnings and potential stale state.
- **Fix:** Moved the early-return below all `useMemo`s so hook order is constant across renders.
- **Files modified:** `frontend/src/components/viz.jsx`
- **Verification:** `npm run check` + `npm run build` green; dev-server smoke (curl 5173) returned 200 with no console errors.
- **Committed in:** `8ae4c7d` (Task 1 commit)
