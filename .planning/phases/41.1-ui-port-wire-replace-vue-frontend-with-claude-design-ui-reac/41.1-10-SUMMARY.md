# Phase 41.1 Plan 10: caffeinate wake-lock + Long-Task TaskBanner

Two compound NRs closed: NR-7 (advisory dies on macOS sleep) and NR-8 (no long-task UX affordance). The fix is twofold — prevent the sleep at the OS level (caffeinate) AND surface the failure mode if prevention misses (health probe + red banner).

## What Closed

| NR | Description | Resolution |
|----|-------------|------------|
| NR-7 | Advisory stuck forever after M1 sleep — Ollama unloaded the orchestrator mid-job. | (a) Backend wraps `synthesize()` in `caffeinate -i` Popen + terminate-in-finally. (b) `GET /api/health/ollama` exposes `{connected, models_loaded}` always-200. (c) Frontend `useOllamaHealth` polls 5s while active; TaskBanner switches to red + Retry on `connected===false`. |
| NR-8 | No way to distinguish "running" from "stuck" during 5+ min advisory or 54-min sim. | New `<TaskBanner>` mounted between topbar-error and SignalWire. Renders nothing when idle; while active shows task name, mm:ss elapsed, OLLAMA CONNECTED + model count (advisory only), and Cancel (sim) / Retry (advisory + dead Ollama). |

## NR-7 — Backend caffeinate

### `_run_advisory_synthesis` diff

The function now spawns a `caffeinate -i` subprocess BEFORE the synthesize block and terminates it in finally:

```python
caffeinate_proc: subprocess.Popen[bytes] | None = None
if shutil.which("caffeinate") is not None:
    try:
        caffeinate_proc = subprocess.Popen(
            ["caffeinate", "-i"],
            stdout=subprocess.DEVNULL,
            stderr=subprocess.DEVNULL,
        )
        log.info("advisory_caffeinate_started", cycle_id=cycle_id, pid=caffeinate_proc.pid)
    except (FileNotFoundError, OSError) as exc:
        log.warning("advisory_caffeinate_unavailable", ...)
        caffeinate_proc = None
else:
    log.warning("advisory_caffeinate_skipped", reason="binary_missing")

try:
    # ... existing synthesize + write-file path ...
finally:
    # ... existing unload_model ...
    if caffeinate_proc is not None:
        try:
            caffeinate_proc.terminate()
            try:
                caffeinate_proc.wait(timeout=2.0)
            except subprocess.TimeoutExpired:
                caffeinate_proc.kill()
            log.info("advisory_caffeinate_terminated", cycle_id=cycle_id)
        except Exception as exc:
            log.warning("advisory_caffeinate_terminate_failed", error=str(exc))
```

Deviation from plan: added a `wait(timeout=2.0)` + `kill()` fallback (not in the original plan source). Rationale: a hung caffeinate handle (rare but observed under memory pressure) could leak file descriptors on the FastAPI worker process. This is a Rule 1 auto-fix (correctness — terminate without wait is incomplete cleanup), not architectural.

### `/api/health/ollama` route

Extended the existing `health.py` with a second endpoint:

```python
@router.get("/health/ollama", response_model=OllamaHealthResponse)
async def health_ollama(request: Request) -> OllamaHealthResponse:
    ollama_client = getattr(app_state, "ollama_client", None)
    if ollama_client is None:
        return OllamaHealthResponse(connected=False, models_loaded=[])
    try:
        ps_response = await ollama_client.raw_client.ps()
        # parse models field (handles both ProcessResponse and dict shape)
        return OllamaHealthResponse(connected=True, models_loaded=names)
    except Exception:
        return OllamaHealthResponse(connected=False, models_loaded=[])
```

**Method choice (recorded per plan output spec):** `raw_client.ps()` — returns CURRENTLY-LOADED models. The alternative `raw_client.list()` returns the full Ollama registry (every pulled model on disk), which would NEVER indicate a sleep-induced unload. `ps()` is Ollama's "running models" probe and matches the NR-7 detection semantics exactly.

The endpoint NEVER raises — every failure path (no client, ps() exception, malformed response) returns 200 with `connected=false`. This lets `useOllamaHealth` poll without try/catch at every tick.

App registration: the health router was already registered in `app.py` line 131 (Phase 32 baseline) — no app.py edit needed.

## NR-8 — Frontend TaskBanner

### Component contract

```tsx
interface TaskBannerProps {
  running: boolean;                  // simulation active
  advisorySynthesizing: boolean;     // any cycle's advisory in flight
  onCancelSim: () => void;           // wired to onStop (simStop)
  onRetryAdvisory?: () => void;      // wired to handleRetryAdvisory
}
```

Returns `null` when neither task is active. While active:
- Background: `--accent` (orange) normally, `--sell` (red) when advisory + Ollama disconnected.
- Layout: task name (700-weight), elapsed mm:ss, Ollama health phrase (advisory only), spacer, Cancel/Retry button.
- ARIA: `role="status"`, `aria-live="polite"` for screen-reader politeness.

### Integration point

Mounted in `app_v2.jsx` between the `connection-banner` and `<SignalWire>` (immediately after the existing reconnect-failed banner block). This places it ABOVE the simulation viz and BELOW the topbar, sharing the chrome strip.

### advisorySynthesizing derivation (banner activation logic)

```jsx
useEffect(() => {
  const isComplete = phase === 'complete' || phase === 'done';
  if (!isComplete || !currentCycleId) {
    setAdvisorySynthesizing(false);
    return;
  }
  if (!hasAdvisoryBeenTriggered(currentCycleId)) return;
  setAdvisorySynthesizing(true);
  const t = window.setTimeout(() => setAdvisorySynthesizing(false), 12 * 60 * 1000);
  return () => window.clearTimeout(t);
}, [phase, currentCycleId]);
```

Three-input gate: phase complete AND known cycleId AND auto-trigger registry mark. The registry-mark check ties the banner to the same source-of-truth used by Plan 09's auto-trigger and the modal's idempotent kick — no duplicate state.

### useOllamaHealth hook

Active-gated 5s poller — only fires while the consumer passes `active=true`. When inactive, the effect cleanup clears the interval AND resets stored health to `null` so consumers can rely on `health === null` meaning "no live signal".

```ts
useEffect(() => {
  if (!active) {
    setHealth(null);
    setError(null);
    return;
  }
  let cancelled = false;
  const tick = () => { /* getOllamaHealth + cancellation guard */ };
  tick();                                        // immediate fire
  const id = window.setInterval(tick, 5000);
  return () => { cancelled = true; window.clearInterval(id); };
}, [active]);
```

Failure path: `.catch` synthesizes `{ connected: false, models_loaded: [] }` so the banner can render the disconnected state without a separate error branch.

### AdvisoryModal Error state + Retry

Replaces the previous one-liner `Advisory failed: …` with a red-bordered card containing message + Retry button:

```jsx
<button onClick={() => {
  setKickError(null);
  if (cycleId) unmarkAdvisoryTriggered(cycleId);
  setKicked(false);
}}>
  Retry
</button>
```

The Retry handler clears (a) the kick-error React state, (b) the module-level registry mark via the new `unmarkAdvisoryTriggered` export, and (c) the `kicked` flag so the existing kick effect re-fires. Backend already pops any stale error entry on the new POST (advisory.py:128-129), so polling resumes cleanly.

### `unmarkAdvisoryTriggered` export

Added to `useAdvisoryAutoTrigger.ts`:

```ts
export function unmarkAdvisoryTriggered(cycleId: string): void {
  advisoryAutoTriggered.delete(cycleId);
}
```

Without this, a single backend failure would lock the cycleId out of the auto-trigger for the rest of the browser session. Both Retry surfaces (TaskBanner + AdvisoryModal) call it.

## Deviations from Plan

**1. [Rule 1 - Bug] Added `kill()` fallback after `wait(timeout=2.0)` TimeoutExpired**
- **Found during:** Task 1 implementation
- **Issue:** Plan source had `terminate() + wait(timeout=2.0)` only. If caffeinate ignored SIGTERM (rare under heavy memory pressure), the `wait` would raise `subprocess.TimeoutExpired` and the FastAPI worker would leak the file descriptor.
- **Fix:** Wrapped `wait()` in inner try/except and call `kill()` on `TimeoutExpired`. Outer try/except still catches all other failures (defensive — never propagate from finally).
- **Files modified:** `src/alphaswarm/web/routes/advisory.py`
- **Commit:** `5d578fa`

**2. [Rule 3 - Blocking] Frontend node_modules absent in worktree clone**
- **Found during:** Task 2 — `npm run check` failed with `tsc: command not found`.
- **Fix:** Ran `npm install` in `frontend/` (94 packages, no vulnerabilities). This is a worktree environment setup, not a code change — no commit; package-lock.json is already tracked at the repo root.

**3. [Rule 1 - Bug] Plan instructed `import { advisoryGenerate } from '../api/advisory'` with the comment "confirm path matches modals.jsx import"**
- **Found during:** Task 2 Step 4
- **Issue:** Plan source had a TODO-style comment but no concrete confirmation. Verified via Read that `frontend/src/api/advisory.ts` exports `advisoryGenerate` and modals.jsx imports it from that path — used the same import path.
- **Fix:** No change needed; just confirmation that the plan path was correct.

**4. [Rule 2 - Critical] Added React 19 JSX type import**

- **Files modified:** `frontend/src/components/TaskBanner.tsx`
- **Commit:** `6284ee2`

**5. [Plan refinement] AdvisoryModal Error block — used `formatApiError(hadError)` instead of `String(hadError?.message || hadError)`**
