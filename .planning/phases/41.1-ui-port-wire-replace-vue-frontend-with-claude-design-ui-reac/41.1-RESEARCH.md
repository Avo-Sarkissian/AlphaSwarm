## Summary

Phase 41.1 is a **stack swap + wiring port**, not a greenfield FE build. The design team exported 12 React/JSX files, a CSS file, and HTML references to `Claude Design UI/AlphaSwarm/`. All 12 components are **pre-shaped against `CONTRACT.md` v0.1** — they already expect the data shapes the backend should emit. The real work is:

1. **Stack swap.** Replace Vue (vue + @vitejs/plugin-vue + vue-tsc) with React 19.2 (react + react-dom + @vitejs/plugin-react + @types/react + @types/react-dom). Keep Vite 6, TypeScript 5.6, D3 libs, dompurify, marked.
2. **Namespace conversion.** The design ships using `window.*` globals (UMD React + in-browser Babel pattern from `v2.html`). Every `window.Icon`, `window.AS_DATA`, `window.Viz`, `window.BracketList`, `window.parent.postMessage` reference must be converted to ES module imports. This is the *only* structural edit permitted under D-06.
3. **Wire mock data to live hooks.** Every reader of `window.AS_DATA.*` (BRACKETS, buildAgents, RATIONALES, HOLDINGS, REPLAY_CYCLES, DATA_SOURCES, SIGNAL_WIRE_SEED, SOURCE_STATS) is a wiring point. Replace with `useWebSocket()`, `useFetch()`, or Context consumption.
4. **Port `useWebSocket` composable to a React hook** preserving reconnect-backoff logic (1s→2s→4s→8s, fail after 3), rationale dedup+cap, phase-idle clear, teardown null'ing `onclose`.
5. **Build a data adapter.** Backend `StateSnapshot` and CONTRACT `StateFrame` **do not match** verbatim — see "Data-Shape Conformance Matrix." An adapter maps backend → CONTRACT shape at the WebSocket boundary.
6. **Live parity verification.** Boot backend + new React UI + old Vue UI side-by-side (different ports), run identical seeds, and confirm per-surface equivalence.

**Primary recommendation:** Execute as a single-wave drop-and-wire operation: (1) preserve old `frontend/` as `frontend-vue-archive/` for parity diffing during the phase, (2) scaffold new React frontend in place, (3) convert globals → ESM, (4) wire hooks, (5) run parity checklist, (6) delete archive when green, (7) update CLAUDE.md.

### Pattern 1: WebSocket → Context → Component

**What:** Single WebSocket connection at app root, distributed via three Contexts (snapshot, rationales, connection). Matches the Vue `provide/inject` pattern (D-10).

**When to use:** Default for all live data in this phase. Use Zustand only if profiling shows context re-renders are a bottleneck.

**Example:**
```tsx
// Source: Ported from frontend/src/composables/useWebSocket.ts (Vue 3 composable)
// Target: frontend/src/hooks/useWebSocket.ts (React 19 hook)

import { useEffect, useRef, useState, useCallback } from 'react';
import type { StateSnapshot, RationaleEntry } from '../types';
import { adaptFrame } from '../adapter/frame';

export function useWebSocket() {
  const [snapshot, setSnapshot] = useState<StateSnapshot | null>(null);
  const [latest, setLatest] = useState<RationaleEntry[]>([]);
  const [all, setAll] = useState<RationaleEntry[]>([]);
  const [connected, setConnected] = useState(false);
  const [reconnectFailed, setReconnectFailed] = useState(false);

  const wsRef = useRef<WebSocket | null>(null);
  const attemptsRef = useRef(0);
  const timerRef = useRef<number | null>(null);
  const mountedRef = useRef(true);

  const connect = useCallback(() => {
    if (!mountedRef.current) return;
    const url = getWsUrl();
    const ws = new WebSocket(url);
    wsRef.current = ws;

    ws.onopen = () => {
      setConnected(true);
      setReconnectFailed(false);
      attemptsRef.current = 0;
    };

    ws.onmessage = (ev) => {
      const raw = JSON.parse(ev.data);
      const frame = adaptFrame(raw);           // backend → CONTRACT shape
      setSnapshot(frame);
      // rationale dedup by agent_id:round_num, cap 20
      if (raw.rationale_entries?.length) {
        setLatest(raw.rationale_entries);
        setAll((prev) => dedupCap(prev, raw.rationale_entries, 20));
      }
      if (frame.phase === 'idle') {
        setAll([]);
        setLatest([]);
      }
    };

    ws.onclose = () => {
      setConnected(false);
      if (!mountedRef.current) return;
      attemptsRef.current += 1;
      if (attemptsRef.current >= 3) {
        setReconnectFailed(true);
        return;
      }
      const backoff = 1000 * Math.pow(2, attemptsRef.current - 1); // 1s, 2s, 4s
      timerRef.current = window.setTimeout(connect, backoff);
    };

    ws.onerror = () => ws.close();
  }, []);

  useEffect(() => {
    mountedRef.current = true;
    connect();
    return () => {
      mountedRef.current = false;
      if (timerRef.current) window.clearTimeout(timerRef.current);
      if (wsRef.current) {
        wsRef.current.onclose = null; // critical: prevent reconnect after teardown
        wsRef.current.close();
      }
    };
  }, [connect]);

  return { snapshot, latest, all, connected, reconnectFailed };
}
```

### Pattern 2: Design's Pure-SVG viz.jsx (not d3-force)

**What:** The design's `viz.jsx` does NOT use d3-force simulation. It uses `useMemo` over a layout function that returns node positions deterministically (radial, grid, or pseudo-force math) and renders SVG `<circle>` + `<path>` directly.

**When to use:** Always (per D-06, verbatim drop).

**Implication:** Current Vue `ForceGraph.vue`'s `forceSimulation`/`forceManyBody`/`forceCollide`/`forceLink` code is **not ported**. It is replaced by `viz.jsx`'s deterministic layout. This is a behavior change — user explicitly accepted this via D-06 ("verbatim drop").

**Verification risk:** The new "force" layout in `viz.jsx` must be confirmed to produce recognizable clusters when 100 agents + edges are rendered against live data. If the deterministic layout looks wrong for live data, we fix it by **adjusting layout math in `viz.jsx`**, not by reintroducing d3-force (which would violate D-06's structural-preservation intent).

**What:** Design files use `window.*` globals. Conversion is the one structural edit allowed under D-06.

2. Replace `const { Icon } = window;` with `import { Icon } from './icons';`
3. Replace `window.AS_DATA.*` with live-hook consumption OR `../mocks/data` import (mocks only used outside render path).
4. Replace `Object.assign(window, { Foo, Bar });` with `export { Foo, Bar };`
5. Strip `window.parent.postMessage({ type: '__activate_edit_mode' }, '*')` calls (Claude Artifacts bridge).

### Pattern 4: Async Job Polling Hook (D-12)

**What:** Report and advisory endpoints return 202 + task id; UI polls GET until 200.

**Example:**
```tsx
// Source: Ported from frontend/src/components/ReportViewer.vue pollReport() pattern
export function usePolling<T>(
  url: string | null,
  intervalMs = 3000,
  maxAttempts = 200
) {
  const [data, setData] = useState<T | null>(null);
  const [error, setError] = useState<string | null>(null);
  const [status, setStatus] = useState<'idle' | 'pending' | 'done' | 'error'>('idle');

  useEffect(() => {
    if (!url) { setStatus('idle'); return; }
    let cancelled = false;
    let attempts = 0;
    setStatus('pending');

    const tick = async () => {
      if (cancelled) return;
      attempts += 1;
      try {
        const r = await fetch(url);
        if (r.status === 202) {
          if (attempts >= maxAttempts) { setError('timeout'); setStatus('error'); return; }
          setTimeout(tick, intervalMs);
          return;
        }
        if (!r.ok) { setError(`${r.status}`); setStatus('error'); return; }
        setData(await r.json());
        setStatus('done');
      } catch (e) {
        setError(String(e));
        setStatus('error');
      }
    };
    tick();
    return () => { cancelled = true; };
  }, [url, intervalMs, maxAttempts]);

  return { data, error, status };
}
```

### Anti-Patterns to Avoid

- **Re-introducing d3-force after dropping `viz.jsx`.** Violates D-06. If deterministic layout needs tuning, tune the math in `viz.jsx`, don't swap libraries.
- **Creating a shared hook that returns the entire snapshot.** Causes every consumer to re-render at 5Hz. Split into granular selectors (context slices OR Zustand selectors).
- **Storing rationales in a ref and imperatively updating DOM.** Breaks React's rendering model; the Vue equivalent worked because Vue's reactivity hid the mutation. In React, use `useState` with a cap-+-dedupe reducer.
- **Running Babel in-browser (design's v2.html pattern).** Vite compiles JSX/TSX. Remove `@babel/standalone` references; remove `type="text/babel"` from `<script>` tags (in the stripped `index.html` there will be none).
- **Keeping `@vitejs/plugin-vue` "just in case."** Vite will emit a warning on unused plugins; drop it.
- **Inline `<script>` tags in `index.html` that reference `window.*` globals.** The ESM conversion makes them unnecessary; dead code after port.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| WebSocket reconnect with exponential backoff | Custom state machine | **Port existing `useWebSocket.ts` logic** (proven, has teardown-safety fix from commit `10d739e`) | The Vue composable already handles 3-attempt cap, null-before-close, phase-idle clear. Rewriting risks re-introducing B11 (the teardown bug fixed in quick task 260416-trw). |
| Markdown rendering with sanitization | Hand-parse markdown | `marked` + `dompurify` | Already in package.json; hand-rolled markdown parsers always have edge cases. |
| SVG layout algorithm | Custom force math | **Use `viz.jsx`'s deterministic layout verbatim** | D-06 locks it; re-implementing would violate "verbatim drop." |
| Global state for 5Hz snapshots | Redux / MobX / custom bus | **React Context first** → Zustand fallback (D-10) | Context handles it for <100 updates/sec when providers are split; if profiling shows a problem, Zustand ships with `useSyncExternalStore` semantics. |
| Long-job polling | Bespoke setInterval loop per modal | **Shared `usePolling<T>` hook** | The advisory, report, and (future) interview-transcript modals all share the 202→200 pattern. |
| JSON-shape adaptation | Inline mapping in every component | **Single `adapter/frame.ts`** at WS boundary | CONTRACT.md vs backend shape is a known conformance gap (see matrix below); components must see CONTRACT shape only. |
| Connection-status banner | Per-component ws state | **`ConnectionContext`** (D-10) | Matches Vue `provide/inject` pattern for `connected` + `reconnectFailed`. |
| REST client | Raw fetch everywhere | **Thin typed wrappers under `src/api/*`** | Keep `fetch` (don't add axios) but centralize retry/error mapping. |

**Key insight:** The design-team's export already solves layout, theming, and most interaction logic. **The planner's primary engineering concern is the wiring surface (adapter + hooks + context), not component code.** Rewriting the design components would waste effort and violate D-06.

## Common Pitfalls

### Pitfall 1: `tsconfig.app.json` still has `jsx: "preserve"`
**What goes wrong:** Build emits errors like "JSX element has no corresponding closing tag" or types fail on `<App/>` usage.
**Why it happens:** Vue repo was configured with `"jsx": "preserve"` (Vue handles JSX itself) and `"include": ["src/**/*.vue", ...]`. After the stack swap, `tsc --noEmit` will still look for Vue files.
**How to avoid:** Set `"jsx": "react-jsx"`, remove `*.vue` from `include`, add `*.tsx`/`*.jsx`, and set `"allowJs": true` during the transitional period where design files are still `.jsx`.
**Warning signs:** `frontend:check` (or `tsc -b --noEmit`) fails at step 1 after dependency swap.

### Pitfall 2: React 19 Strict Mode double-invokes `useEffect` in dev
**What goes wrong:** `useWebSocket` opens two connections on mount; reconnect counter jumps to 2 before any real failure; snapshot state flickers.
**Why it happens:** React 19 (like 18) intentionally double-invokes setup/cleanup in `<StrictMode>` in dev only.
**How to avoid:** Ensure teardown is idempotent (null `onclose`, close only if `readyState !== CLOSED`, clear timer). Port the existing Vue teardown logic verbatim — it was already hardened in quick task `260416-trw` (commit `10d739e`, bug B11).
**Warning signs:** In dev: "WebSocket connection closed" + immediate reopen on initial page load; reconnect counter > 0 without network error.

### Pitfall 3: 5Hz × 100 agents re-render storm
**What goes wrong:** FPS drop in the force graph; KPI strip flickers; rationale feed janky.
**Why it happens:** A single `SnapshotContext` value change re-renders every consumer. At 5Hz, 100-node SVG updates combined with rationale feed re-render can exceed 16ms frame budget.
**How to avoid (in order of preference):**
1. Split context into `SnapshotContext` (slow-changing phase/round) + `AgentsContext` (100 nodes) + `TelemetryContext` (tps/mem) — consumers subscribe only to what they need.
2. Use `React.memo` on `viz.jsx`, `KpiStrip`, `BracketList`, `RationaleFeed` with custom equality.
3. Fall back to Zustand per D-10 with selector-based subscriptions (`useStore(s => s.telemetry.tps)`).
4. If graph is still the bottleneck, render `<circle>` positions via refs + `useEffect` rather than setState (imperative DOM update inside React's rendering contract — used by `viz.jsx`'s `requestAnimationFrame` path).
**Warning signs:** Chrome DevTools "Performance" shows long tasks > 50ms at 5Hz cadence; React DevTools Profiler shows component trees re-rendering every tick.

### Pitfall 4: `window.parent.postMessage` call in app_v2.jsx leaks to host
**What goes wrong:** Dev console warnings about cross-origin postMessage; potentially fires during embed in iframe.
**Why it happens:** Design files were exported from Claude Artifacts which uses a parent-frame edit bridge.
**How to avoid:** Grep for `window.parent.postMessage`, `__activate_edit_mode`, and any iframe-host signaling; delete all matches during conversion. Verified references exist in `app.jsx` and `app_v2.jsx`.
**Warning signs:** Browser console "Failed to execute 'postMessage' on 'Window': ... target origin" in standalone mode.

### Pitfall 5: Design's bracket counts (`data.jsx`) contradict CONTRACT §6 and backend persona config
**What goes wrong:** Mock mode (tweaks panel) shows wrong bracket distributions; parity test fails because design's `BRACKETS` constant doesn't match backend emission.
**Why it happens:** `data.jsx` has a BRACKETS array with counts summing to 100 in one distribution; CONTRACT §6 requires a *different* 100-sum distribution; the real backend persona loader emits a *third* distribution.
**How to avoid:** Never drive the live render from `data.jsx` BRACKETS. Live UI must read `snapshot.bracket_summaries` which is authoritative (backend-emitted). Reserve `data.jsx` for the tweaks/mock panel only.
**Warning signs:** BracketList renders 10 entries with zero counts because live buckets don't match hardcoded keys.

### Pitfall 6: Phase enum mismatch (backend string vs CONTRACT integer)
**What goes wrong:** `KpiStrip` round indicator shows `undefined` or `"round_1"` instead of `1`/`2`/`3`/`done`.
**Why it happens:** Backend `SimulationPhase` enum: `idle|seeding|round_1|round_2|round_3|complete|replay`. CONTRACT §3 `StateFrame.phase`: `1|2|3|'done'`.
**How to avoid:** In `adapter/frame.ts`, remap `round_1→1`, `round_2→2`, `round_3→3`, `complete→'done'`, `idle|seeding|replay→` a separate discriminant the overlay code (`states.jsx`) inspects.
**Warning signs:** `switch (phase) { case 1: ... }` never matches.

### Pitfall 7: Rationale entries missing fields expected by CONTRACT
**What goes wrong:** Rationale cards show `undefined` for citations/sources/timestamp, or the highlight regex crashes on missing `text`.
**Why it happens:** Backend emits `{agent_id, signal, rationale, round_num}`; CONTRACT §2.3 expects `{id, ts, agentId, round, signal, confidence, text, citations[], sources[]}`.
**How to avoid:** In `adapter/frame.ts`, synthesize `id = ${agent_id}:${round_num}`, `ts = Date.now()`, `agentId = agent_id`, `round = round_num`, `text = rationale`, `confidence = 0` (or read from agent_states[agent_id].confidence), `citations = []`, `sources = []`. Design must tolerate empty citations/sources (confirm by reading `panels.jsx` RationaleFeed — it reads `r.citations || []`).
**Warning signs:** `TypeError: Cannot read properties of undefined (reading 'map')` in RationaleFeed.

### Pitfall 8: Edges endpoint is REST, not streamed via WebSocket
**What goes wrong:** Design `viz.jsx` expects `StateFrame.edges[]`; backend does NOT put edges in the snapshot.
**Why it happens:** Backend edges are per-cycle/per-round and fetched via `GET /api/edges/{cycle_id}?round=N`.
**How to avoid:** Add an `useEdges(cycleId, round)` hook that refetches on `round` change; feed the result into `SnapshotContext` adapter as `frame.edges`. Do NOT block rendering on edge fetch — initial render must show nodes without edges.
**Warning signs:** Viz shows nodes but no connecting lines; or nodes+edges render only after a delay that doesn't match the 5Hz cadence.

### Pitfall 9: SignalWire + DataSources have no backend support
**What goes wrong:** Design `v2.jsx` SignalWire ticker and DataSources modal consume `window.AS_DATA.SIGNAL_WIRE_SEED` / `DATA_SOURCES` / `SOURCE_STATS`. No `/api/wire` or `/api/sources` exists.
**Why it happens:** CONTRACT §2.4 (WireEvent) and §2.5 (DataSource/SourceStats) are unimplemented.
**How to avoid:** Render SignalWire from a static fixture in `mocks/data` in the immediate port. Flag as "deferred backend work" in Open Questions. Do NOT block the port on inventing these endpoints. User deferred this explicitly (CONTEXT Deferred Ideas implicit).
**Warning signs:** Plan assumes WebSocket contains wire events — it doesn't.

### Pitfall 10: `frontend/dist` and `frontend/node_modules` not cleaned
**What goes wrong:** FastAPI serves stale Vue `index.html`; `npm install` finds old vue entries in lockfile; TypeScript picks up old `.vue.d.ts`.
**Why it happens:** Vite dev server doesn't rebuild `dist`; lockfile holds vue pins.
**How to avoid:** `rm -rf frontend/node_modules frontend/package-lock.json frontend/dist` after package.json rewrite; then `npm install`. Explicit task in plan.
**Warning signs:** After port, backend still serves Vue UI when accessed at `http://localhost:8000`.

### Pitfall 11: Vite dev proxy port mismatch with module-level app instance
**What goes wrong:** Dev flow `vite` on :5173 → `fetch('/api/health')` proxied to `:8000` works only if FastAPI is actually bound to :8000.
**Why it happens:** Recent commit `fd0906a` added module-level `app = create_app()` so `uvicorn app:app` works, but `python -m alphaswarm.cli ui start` and raw `uvicorn src.alphaswarm.web.app:app` both still bind :8000 by default.
**How to avoid:** Keep `vite.config.ts` proxy `{ '/api': 'http://localhost:8000', '/ws': { target: 'ws://localhost:8000', ws: true } }` unchanged. Verify dev doc in phase output says "start backend first on :8000, then `cd frontend && npm run dev`."
**Warning signs:** 500/ECONNREFUSED on every API call during development.

### Pitfall 12: CSS variable collision between design and current Vue
**What goes wrong:** Old Vue `types.ts` defines `SIGNAL_COLORS.buy = '#22c55e'`; new design styles.css defines `--buy: #42d690`. If both are loaded, components reading via `types.ts` constants differ from components reading CSS vars.
**Why it happens:** Vue FE hardcoded colors in TS; design drives everything from CSS custom properties.
**How to avoid:** During port, delete hardcoded hex constants from `types.ts`. Components should read computed CSS `var(--buy)` / `var(--sell)` / `var(--hold)` for non-SVG paths, or re-read the design's hex values (`#42d690`, `#ff5b6b`, `#6b7480`) in TS constants if SVG fill requires a literal.
**Warning signs:** Two UI elements showing different shades of green for BUY.

### Example F: Frame adapter (backend → CONTRACT)

```ts
// Source: New file, frontend/src/adapter/frame.ts
import type { StateFrame } from '../types';

type BackendSnap = {
  phase: 'idle'|'seeding'|'round_1'|'round_2'|'round_3'|'complete'|'replay';
  round_num: number;
  agent_count: number;
  agent_states: Record<string, { signal: 'BUY'|'SELL'|'HOLD'; confidence: number }>;
  elapsed_seconds: number;
  tps: number;
  governor_metrics: { memory_percent: number; [k: string]: unknown };
  rationale_entries: { agent_id: string; signal: string; rationale: string; round_num: number }[];
  bracket_summaries: {
    bracket: string; display_name: string;
    buy_count: number; sell_count: number; hold_count: number;
    total: number; avg_confidence: number; avg_sentiment: number;
  }[];
};

export function adaptFrame(raw: BackendSnap): StateFrame {
  const phase =
    raw.phase === 'round_1' ? 1 :
    raw.phase === 'round_2' ? 2 :
    raw.phase === 'round_3' ? 3 :
    raw.phase === 'complete' ? 'done' :
    raw.phase; // idle | seeding | replay passed through for overlay logic

  const agents = Object.entries(raw.agent_states).map(([id, s]) => {
    const bracket = deriveBracket(id);              // id prefix → bracket key
    return {
      id,
      bracket,
      bracketDisplay: bracketDisplayName(bracket),
      signal: s.signal,
      confidence: s.confidence,
      flipped: false,                               // backend has no flip history
      roundLastSpoke: raw.round_num,                // best-effort
      thinking: false                               // backend has no thinking flag
    };
  });

  const { buy, sell, hold } = tally(raw.agent_states);
  const consensus = { buy, sell, hold, dominant: dominantOf(buy, sell, hold) };

  return {
    ts: Date.now(),
    phase,
    running: phase === 1 || phase === 2 || phase === 3 || raw.phase === 'seeding',
    agents,
    consensus,
    telemetry: {
      tps: raw.tps,
      memMb: Math.round(raw.governor_metrics.memory_percent), // stub: % not MB; document
      slotsUsed: 0,    // backend does not emit; stub
      slotsMax: 8,     // stub from config default; wire to /api/health if needed
      elapsedSec: raw.elapsed_seconds
    },
    edges: []         // filled separately by useEdges hook
  };
}
```

## Open Questions

1. **Unit for `memMb` in KPI strip — MB or %?**
   - What we know: Backend emits `memory_percent` (0–100). Design `KpiStrip` label reads "MEM" and the CONTRACT field is named `memMb`.
   - What's unclear: Whether to display `"91%"` or `"43.7GB"` or both.
   - Recommendation: Start with `"{memory_percent}%"`. If user prefers MB, compute `Math.round(48 * percent / 100)` and append " GB". Flag on first demo.

2. **`telemetry.slotsUsed` and `slotsMax` real source.**
   - What we know: Backend does not emit these in the snapshot.
   - What's unclear: Whether to expose them via `/api/health` extension or hardcode from config.
   - Recommendation: Stub to `{used: 0, max: 8}` in adapter; later wave can add a `GET /api/telemetry/slots` endpoint if needed.

3. **Agent-ID → bracket derivation in adapter.**
   - What we know: `states.jsx` hardcodes a prefix map `{Quants:'Q', Degens:'D', Sovereigns:'S', Macro:'M', Suits:'U', Insiders:'I', Agents:'A', 'Doom-Posters':'X', 'Policy Wonks':'P', Whales:'W'}`.
   - What's unclear: Whether backend emits IDs in this exact format (e.g., `Q-01`, `D-05`).
   - Recommendation: **Before starting the port**, snapshot one live backend frame and confirm ID prefixes. If mismatch, the derivation table moves to `adapter/frame.ts` and is updated to match backend. (This is a 30-second verification step, blocking for the adapter task.)

4. **Agent `flipped` / `roundLastSpoke` / `thinking` fields.**
   - What we know: CONTRACT expects them; backend does not emit them.
   - What's unclear: Whether UI visibly degrades without them.
   - Recommendation: Stub all three in adapter (`flipped: false`, `roundLastSpoke: currentRound`, `thinking: false`). If post-parity review shows visible regressions (e.g., no "flip" animation), wire as a Phase 41.2 follow-up via a backend event stream.

5. **D3-force removal.**
   - What we know: Design doesn't use `d3-force`; Vue did.
   - What's unclear: Keep or remove from `package.json`.
   - Recommendation: Keep during port (belt-and-suspenders), remove in phase-exit cleanup task after parity verified.

6. **Settings modal backend binding.**
   - What we know: `settings.jsx` is fully local-state; no `/api/settings` endpoint exists.
   - What's unclear: Whether user expects "concurrent slots" setting to persist to backend config.
   - Recommendation: Ship settings as client-state-only for 41.1. Defer persistence to a later phase.

7. **Old Vue archive retention policy.**
   - What we know: Clean swap per D-02.
   - What's unclear: Whether to delete `frontend-vue-archive/` at phase-end or keep for N weeks.
   - Recommendation: Keep for 1 release cycle (until v6.0 ships and stabilizes), then delete. Add to phase execution notes.
