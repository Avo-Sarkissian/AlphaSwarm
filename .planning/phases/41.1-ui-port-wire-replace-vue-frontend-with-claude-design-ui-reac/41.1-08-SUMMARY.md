## New Cypher Query

```cypher
MATCH (src:Agent)-[infl:INFLUENCED_BY {cycle_id: $cycle_id, round: $round}]->(tgt:Agent)
RETURN
    src.id AS source_id,
    tgt.id AS target_id,
    infl.weight AS weight
```

Parameter-bound (no string interpolation → injection-safe). Returns per-row tuples, not aggregates — distinguishes from `read_influence_leaders` which uses `sum(infl.weight)` + `count(src)`.

## Decisions Made

- Frontend adapts to EdgeItem schema (source_id / target_id) — NOT vice versa. Renaming the response shape would break every existing tool that hits /api/edges; the frontend was the only side reading the wrong keys.
- Added `read_influence_edges` as a sibling method (not a refactor of `read_influence_leaders`). The two queries return different shapes — leaders aggregates by target with sum + count; edges returns per-row tuples — so reuse would force a flag-laden polymorphic method. Two clean methods is simpler.
- Round 0 short-circuits with `if round_num == 0: return []` BEFORE opening a Neo4j session. Round 0 INFLUENCED_BY edges never exist (writes happen during round 1..3 finalization), so the query would always return empty; skipping the round-trip eliminates a needless DB call on the legitimate "pre-simulation" state.
- Weight is accepted into BackendEdge but ignored by the force layout. Plan 02's force-layout edge layer doesn't consume edge weights yet; deferring weight-based styling keeps this fix minimal and shippable.
