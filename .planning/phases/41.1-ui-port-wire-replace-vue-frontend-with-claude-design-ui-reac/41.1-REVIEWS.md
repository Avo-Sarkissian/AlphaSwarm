# Cross-AI Plan Review — Phase 41.1

## Gemini Review

# Phase 41.1 Plan Review: UI Port & Wire (Vue → React)

## 1. Summary
The plan for Phase 41.1 is exceptionally well-structured, treating a high-risk stack swap (Vue to React) as a surgical procedure rather than a rewrite. By archiving the existing Vue frontend (`frontend-vue-archive/`) and configuring it to run on a separate port (:5174), the plan builds in a robust safety net for side-by-side parity verification. The multi-wave approach correctly prioritizes infrastructure setup (Waves 0–1) before surface-level wiring (Wave 2), and the decision to accept specific known regressions (KR-41.1-01..09) up-front keeps the scope focused on implementation parity. The use of a dedicated adapter layer (`adaptFrame`) to bridge the gap between the backend's existing schema and the design export's `CONTRACT.md` requirement is a critical architectural win that avoids unnecessary backend churn.

## 2. Strengths
*   **Archival Strategy:** Moving the Vue frontend to a sibling directory with a renamed package and distinct port is an industry-best-practice for manual UAT and debugging during a framework migration.
*   **Schema Adaptation:** Identifying that the backend and design contract mismatch and building an explicit `adaptFrame` function at the WebSocket boundary is the correct way to insulate components from data-shape volatility.
*   **B11 Regression Guard:** Explicitly carrying over the WebSocket teardown logic (nulling `onclose` on unmount) from the repository's bug-fix history shows high attention to detail.
*   **Sound Parallelization:** Splitting Wave 2 into 03 (Main Dashboard) and 04 (Modals/Overlays) is clean because there is zero file overlap, ensuring no merge conflicts or race conditions during execution.
*   **Security Consistency:** Preserving and enforcing the `dompurify` + `marked` pipeline for all markdown paths (rationales, reports, advisories) maintains the project's security posture.
*   **Diagnostic Greps:** The inclusion of specific grep commands (Task 1.5 in Plan 02) to catch residual globals or Claude Artifacts bridge calls is a great operational safety gate.

## 3. Concerns
*   **Re-render Performance (Severity: MEDIUM):** Updating a single `SnapshotContext` at 5Hz with 100 agents might trigger massive re-render trees in React. While Wave 1 research acknowledges this and suggests Zustand as a fallback (D-10), the plans don't explicitly include a profiling step or specific memoization instructions for the 100-node graph components.
*   **Visual "Feel" of viz.jsx (Severity: LOW):** The design's `viz.jsx` uses deterministic math instead of `d3-force`. While user-accepted (KR-41.1-01), the "dead" look of a non-physics graph might be jarring compared to the previous Vue implementation. The plan relies entirely on manual eyeballing in Wave 3.
*   **Edge Data Sync (Severity: LOW):** `useEdges` is a separate REST call. At a 5Hz refresh rate, there is a risk of edges appearing "out of sync" or flickering if the WebSocket frame for a new round arrives significantly before the REST response for that round's edges.
*   **Top Influencers Approximation (Severity: LOW):** KR-41.1-09 substitutes `confidence` for `influence`. This is acceptable for a port, but might hide bugs if the user relies on specific influence-toplogy reporting.

## 4. Suggestions
*   **Context Splitting:** Consider splitting `SnapshotContext` into `MetadataContext` (phase, telemetry, consensus) and `AgentsContext` (the 100-agent array) in Wave 1. This would prevent the KPI strip and round indicators from re-rendering the entire 100-node SVG tree on every tick if only tokens/sec changed.
*   **`useCurrentCycle` Robustness:** Ensure the hook clears the `cycleId` state correctly when `snapshot.phase` returns to `idle`, preventing modals from attempting to fetch data for the previous cycle.
*   **Adapter Debugging:** Add a `console.debug` in `adaptFrame` (guarded by `import.meta.env.DEV`) that logs the first frame received, to help quickly identify any unmapped fields during initial Wave 1/2 integration.
*   **CSS Variable Audit:** In Wave 1 Task 1, do a quick grep for remaining hardcoded hex colors in the design JSX files and replace them with the corresponding `--buy/--sell/--accent` variables from `styles.css` to ensure full theme consistency.

## 5. Risk Assessment
**Risk Level: LOW-MEDIUM**

The primary risks are **performance-related** (5Hz updates for a 100-node tree in a single React Context) and **perceptual** (loss of D3 physics transitions). However, the risk of **functional breakage or data loss is extremely low** due to the archival strategy, the explicit adapter layer, and the comprehensive 28-row parity matrix. The architectural separation of concerns (adapter/hooks/contexts) is robust and follows modern React idioms.

**Verdict:** The plans are ready to execute. Wave 2 parallelization is sound. Archival strategy is top-tier. Proceed with Plan 01.

## Codex Review

## Overall Assessment
The plans are thorough and show strong intent: archive first, build interfaces before wiring, split visible surfaces from modal/history work, and close with human UAT. However, the current plan set is higher risk than it looks. Several items contradict locked decisions or each other, especially endpoint names, accepted-regression count, Wave 03/04 parallel independence, mock-data policy, and API polling semantics. I would not execute these plans as-is. They need a tightening pass before implementation.

## Cross-Plan Issues

**Strengths**
- Clear wave ordering: scaffold → interfaces → wiring → UAT.
- Good emphasis on preserving the Vue archive for parity checks.
- Strong awareness of WebSocket cadence, reconnect handling, markdown sanitization, and local-first constraints.
- The adapter boundary is the right architectural move because backend `StateSnapshot` and design `StateFrame` do not match.
- UAT is explicit and grounded in a 28-row matrix.

**Concerns**
- **HIGH:** Accepted regressions conflict. The phase says 8 known regressions `KR-41.1-01..08` are accepted upfront, but the plans introduce `KR-41.1-09`. That is not a minor bookkeeping issue; it changes the phase acceptance contract.
- **HIGH:** Endpoint naming is inconsistent. Context says `/api/simulation/start|stop|shock`; plans repeatedly use `/api/simulate/start|stop|shock`; status uses `/api/simulation/status`. This must be verified and normalized before execution.
- **HIGH:** Plans 03 and 04 are not actually parallel-safe. Plan 04 depends on `simShock` from `api/simulation.ts`, which Plan 03 creates. Plan 04 also may edit `App.tsx` or `app_v2.jsx`, while Plan 03 owns `app_v2.jsx`.
- **HIGH:** `apiFetch` throws plain `Error` without `status`, but report/advisory polling checks `e.status === 404`. The 404-as-pending contract will not work.
- **HIGH:** `usePolling(() => reportFetch(cycleId), ...)` with an inline function will restart the effect every render unless callers memoize the callback. This can create a polling loop reset/storm.
- **MEDIUM:** Multiple plans say "verbatim design drop," but later tasks delete columns, replace modal bodies, remove summary strips, and alter lifecycle components. Some edits are necessary, but the plan should distinguish "verbatim visual structure" from "verbatim source."
- **MEDIUM:** The plans rely heavily on grep-based acceptance. Greps are useful, but several checks are brittle or give false confidence.

**Suggestions**
- Freeze a single regression register before implementation. Either remove `KR-41.1-09`, fold it into an existing accepted KR, or get explicit user approval for a ninth accepted regression.
- Verify actual backend routes once and put the canonical route table in Plan 02 interfaces.
- Move shared API wrappers used by both 03 and 04 into Plan 02, especially `api/simulation.ts`.
- Fix `apiFetch` to throw an error type that includes `status`, `body`, and `path`.
- Change `usePolling` to accept `{ key, fetchFn }` or require `useCallback` at every call site.
- Add a short "parallel contract" table showing Plan 03 and Plan 04 write sets and shared read-only dependencies.

**Risk Assessment: HIGH**
The strategy is sound, but several concrete plan details will cause compile/runtime failures or workflow contention if executed directly.

## 41.1-01 Plan Review

### Summary
Plan 01 is a good scaffold/archive wave and mostly does the right foundational work. The archive-first approach is correct for parity. The main issues are reproducibility of the Vue archive, local-first violations from remote fonts, and missing Node type dependencies.

### Strengths
- Archives Vue before replacing `frontend/`, preserving rollback and side-by-side parity.
- Keeps new React scaffold minimal and defers component complexity.
- Preserves Vite proxy for `/api` and `/ws`.
- Includes a human checkpoint before proceeding.
- Explicitly cleans stale `dist` and `node_modules`.

### Concerns
- **HIGH:** `index.html` loads Google Fonts from `fonts.googleapis.com` / `fonts.gstatic.com`. That violates the local-first posture unless explicitly accepted.
- **HIGH:** `tsconfig.node.json` includes `"types": ["node"]`, but `@types/node` is not listed in devDependencies. `npm run check` may fail.
- **MEDIUM:** Deleting `frontend-vue-archive/package-lock.json` weakens parity reproducibility. The old Vue UI should run with the exact dependency lock used before the port.
- **MEDIUM:** Wave 0 says "known regressions: none," but replacing the live UI with a placeholder is a temporary functional regression if served from FastAPI between waves.
- **LOW:** Verification commands are somewhat brittle, especially checks around an "empty" recreated `frontend/`.

### Suggestions
- Self-host fonts, vendor them, or use system fonts until the local-first policy explicitly allows external font CDNs.
- Add `@types/node` to devDependencies.
- Preserve the archive lockfile where possible; delete only `node_modules` and `dist`.
- Label the placeholder state as an intentional intermediate wave gap, not "no regression."
- Use `grep -F` for literal checks like `React + TypeScript`.

### Risk Assessment: MEDIUM
Good foundation, but remote fonts and missing `@types/node` are likely to cause policy or build issues.

## 41.1-02 Plan Review

### Summary
Plan 02 has the right architecture: adapter, WebSocket hook, contexts, typed API client, and ESM conversion. It is also the riskiest plan because it tries to mechanically convert 12 design files while simultaneously enforcing mock-removal rules that conflict with the stated Wave 1 end state.

### Strengths
- Adapter boundary is correct and necessary.
- WebSocket hook preserves important behaviors: reconnect, teardown safety, rationale cap, idle clear.
- Context split is a reasonable first pass.
- Explicitly strips Claude Artifacts `postMessage` bridge.
- Identifies known backend/frontend shape gaps instead of hiding them.

### Concerns
- **HIGH:** Task 1 says live render paths must not read mocks, Task 1.5 expects only `tweaks.jsx` and `v2.jsx` to import mocks, but Task 2 says AppShell still renders with mock data until Wave 2. Those cannot all be true.
- **HIGH:** The Task 1.5 browser smoke test gives false confidence because the converted JSX files are not imported yet. Runtime `ReferenceError`s will not surface.
- **HIGH:** `apiFetch` does not preserve HTTP status, breaking Plan 04 polling.
- **HIGH:** The WebSocket hook can crash on malformed-but-valid JSON if required fields like `agent_states` are absent. `adaptFrame(raw)` needs a try/catch or runtime guard.
- **MEDIUM:** Banning all `window.*` in components is too broad. Ban `window.AS_DATA`, `window.React`, and `window.parent`; legitimate browser APIs may be needed.
- **MEDIUM:** Rationale citations/sources are stubbed empty, but UAT later expects citation links. This is an undocumented regression.
- **MEDIUM:** Backoff comments and code disagree. With `MAX_ATTEMPTS = 3` and `>=`, the hook schedules only two retries after the initial close.
- **MEDIUM:** Three contexts may still re-render too much at 5Hz if `SnapshotContext` carries the whole frame.

### Suggestions
- Split Plan 02 into "ESM conversion" and "live infrastructure," or import all converted components into a temporary smoke harness before Task 1.5.
- Make mock policy explicit: Wave 1 may compile with mocks, Wave 2 removes mocks from live render paths.
- Replace `apiFetch` errors with a typed `ApiError`.
- Add defensive validation in `adaptFrame` and catch adapter failures inside `onmessage`.
- Add a `Rationale citations unavailable` KR or wire citations from backend if possible.
- Consider `TelemetryContext`, `AgentsContext`, and `BracketContext` before reaching for Zustand.

### Risk Assessment: HIGH
The architecture is good, but the internal contradictions and broken API error semantics make execution risky.

### Summary

### Concerns

- **HIGH:** `api/simulation.ts` is created here, but Plan 04 depends on `simShock`; this breaks parallel execution.
- **HIGH:** `viz.jsx` fetches edges locally through `useEdges`, but `app_v2.jsx` reads `snapshot.edges.length`. Since `snapshot.edges` remains `[]`, active-edge count will be wrong.
- **HIGH:** `KR-41.1-09` is a ninth regression. The phase says only 8 are accepted upfront.
- **MEDIUM:** `useCurrentCycle` depends on status polling because snapshot lacks `cycle_id`. This is fragile and can go stale across cycles.

- Add `cycleId` and `roundNum` to `StateFrame` in the adapter if backend has them, rather than polling status from the graph.

- Resolve `KR-41.1-09` before execution.

### Risk Assessment: HIGH
