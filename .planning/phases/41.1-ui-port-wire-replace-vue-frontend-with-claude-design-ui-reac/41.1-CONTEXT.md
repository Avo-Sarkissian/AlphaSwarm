# Phase 41.1: UI Port & Wire — Replace Vue Frontend with Claude Design UI React Assets - Context

**Gathered:** 2026-04-20
**Status:** Ready for planning
**Source:** Direct conversation (user provided locked decisions in-session)

<domain>
## Phase Boundary

**In scope:**
- Swap current Vue 3 + TS frontend at `frontend/` with the React/JSX design export at `Claude Design UI/AlphaSwarm/`.
- Adopt **React + Vite + TypeScript** as the new frontend stack (was Vue 3).
- Drop the design's 12 JSX files, HTML, and CSS in verbatim — **no design modifications**.
- Wire the existing FastAPI `/ws/state` WebSocket broadcaster (5Hz `StateFrame` envelope) into the new React components.
- Wire existing REST endpoints (interview, replay, advisory, report, holdings, edges, health, simulation) to the new React surfaces.
- Verify live-data parity — every stat/panel that rendered in the old Vue UI renders in the new React UI with the same data source.
- Update `CLAUDE.md` Technology Stack section: "Vue 3 + TypeScript" → "React + TypeScript".

**Out of scope:**
- Any design changes (colors, layout, copy, motion, component structure) — user explicitly confirmed the design is final.
- Changes to the FastAPI backend WebSocket schema, broadcaster cadence, or REST route shapes.
- New features beyond what is visible in the Claude Design export.
- Migration tooling or rollback scripts (clean swap, no dual-stack).

</domain>

<decisions>
## Implementation Decisions

### Stack
- **D-01 — Adopt React + Vite + TypeScript.** User explicitly chose "Option A (adopt React)" over translating JSX → Vue. Reason: fastest port, preserves design 1:1, WebSocket wiring is the only real engineering.
- **D-02 — Drop Vue entirely.** The `frontend/` directory is replaced wholesale. No dual-stack.
- **D-03 — Keep Vite as bundler.** Both current Vue and new React use Vite — keep `vite.config.ts`, swap plugin from `@vitejs/plugin-vue` to `@vitejs/plugin-react`.
- **D-04 — TypeScript stays.** The design export is `.jsx` (plain JS), but types are required per current repo convention. **Claude's Discretion:** whether to rename `.jsx` → `.tsx` immediately or port incrementally.

### Asset Placement
- **D-05 — Design folder stays at repo root.** `Claude Design UI/AlphaSwarm/` is not moved. The phase reads assets from this path and copies them into `frontend/` during the port.
- **D-06 — Verbatim drop of JSX files.** All 12 `src/*.jsx` files copied into `frontend/src/` without structural changes. Only edits allowed are: (a) adding WebSocket/REST wiring imports, (b) replacing any mock/fixture data imports with live hooks.
- **D-07 — Verbatim drop of HTML + CSS.** `styles.css` → `frontend/src/styles.css`; `v2.html`/`index.html`/`Motion Spec.html` → preserved as reference docs in `frontend/design-reference/` for verification, but the Vite entry uses a stripped `index.html` that mounts React.

### WebSocket Wiring
- **D-08 — Use existing `/ws/state` endpoint unchanged.** The backend broadcaster sends the `StateFrame` envelope at 5Hz (see `Claude Design UI/AlphaSwarm/CONTRACT.md` §3). The React port must consume this frame exactly — no backend schema changes.
- **D-09 — React WebSocket hook.** Port the current Vue `useWebSocket` composable (`frontend/src/composables/useWebSocket.ts`) to a React hook `useWebSocket()` in `frontend/src/hooks/`. Preserve reconnect logic, connection state, and snapshot/rationales state.
- **D-10 — State distribution via React Context.** The old Vue `provide/inject` pattern maps to React Context. Create `SnapshotContext`, `RationalesContext`, `ConnectionContext` providers at app root; components consume via `useContext()`. **Claude's Discretion:** use Zustand if context + 5Hz updates cause performance issues.

### REST Wiring
- **D-11 — Preserve all existing REST routes.** Backend has: `/api/interview`, `/api/replay`, `/api/advisory`, `/api/report`, `/api/holdings`, `/api/edges`, `/api/health`, `/api/simulation`. All stay. React components call them via `fetch` or a thin client.
- **D-12 — Polling for long jobs.** Report and advisory endpoints are async task-based; the Vue UI polls until complete. React port preserves the same polling loops.

### Data-Contract Parity
- **D-13 — `CONTRACT.md` is the binding schema.** `Claude Design UI/AlphaSwarm/CONTRACT.md` v0.1 defines the exact data shapes. The React components already expect these shapes. Backend already conforms (user confirmed UI was designed against live backend). The port must NOT reshape backend data.
- **D-14 — Live-data parity verification.** Every KPI, panel, stat, feed, and modal that rendered in the old Vue UI must render in the new React UI with the same values from the same backend endpoint. Use a side-by-side startup check as the exit criterion.

<specifics>
## Specific Ideas

- The design uses D3 force simulation for the 100-agent "mirofish" graph. Current Vue also uses `d3-force`, `d3-scale`, `d3-selection`, `d3-transition`. Keep these deps — they work identically in React.
- Current frontend uses `dompurify` + `marked` for rendering rationale markdown. Keep these.
- `tsconfig.app.json` + `tsconfig.node.json` exist — update for React JSX (jsx: "react-jsx", allowJs: true during transition).
- Verification strategy: run backend, start a simulation seed, confirm every KPI strip value (tokens/sec, memory %, slots, elapsed, consensus %) updates in new React UI identically to old Vue UI.

</specifics>

<known_regressions>
## Known Regressions (Register)

This phase is an implicit-parity UI port. Some behaviors in the Vue baseline are intentionally not re-implemented in the React port, either because the design does not call for them, or because delivering them would push the phase beyond scope. Each such deviation is captured below as a **Known Regression (KR)** — a sanctioned gap that UAT must green-light, rather than a bug.

Every KR below is pre-authorized by the user and carries forward into the planner/executor chain. Plans MUST declare which KRs they touch in their frontmatter `known_regressions` list; VALIDATION.md rows tied to a KR MUST reference it in their acceptance wording; UAT MAY NOT fail a row that matches an authorized KR.

**Reading key:**
- **ID** — Stable identifier (`KR-41.1-NN`).
- **Scope** — Which baseline behavior is omitted or altered.
- **Rationale** — Why it is acceptable for this phase.
- **Exit** — What future work, if any, would close the gap.

| ID | Scope | Rationale | Exit |
|---|---|---|---|
| **KR-41.1-01** | **Force graph layout — d3-force replaced by deterministic SVG.** `viz.jsx` ships the design export's deterministic SVG layout verbatim; `ForceGraph.vue`'s d3-force physics (wiggle, drift) are not ported. Clusters are static. | Design is source of truth (D-06). Replacing d3-force is a design-authored change, not a port regression. | Later UX pass could reintroduce physics if required; backend already publishes the necessary adjacency data. |
| **KR-41.1-02** | **SignalWire + DataSources modals render from static mocks.** Both surfaces read `mocks/wire.ts` `SIGNAL_WIRE_SEED` and `mocks/sources.ts` `DATA_SOURCES` / `SOURCE_STATS` because the backend exposes no signal-wire or source-stats endpoints. | User-authorized at research time (D-06). Mocks are scoped to `src/mocks/*` so production build never ships them (bundle-leak grep in Plan 05). | New `/api/signal-wire` / `/api/sources` endpoints + live fetch in a future phase. |
| **KR-41.1-03** | **Agent animation fields stubbed in adapter.** `flipped`, `roundLastSpoke`, and `thinking` on the per-agent frame shape are stubbed (`0` / `null` / `false`) because the backend `StateFrame` snapshot has no equivalents. | Design-authored animation fields require backend state the WebSocket does not emit. Visual surfaces fall back to neutral variants. | Backend snapshot extension + adapter wire-up in a dedicated UX phase. |
| **KR-41.1-04** | **Telemetry memory is governor percent, not megabytes.** `telemetry.memMb` carries the ResourceGovernor percent (0–100); the telemetry chip label reads "%" rather than "MB". MemoryPausedState overlay literals (43.7 / 48.0 GB) follow the same contract. | Backend broadcaster emits `memory_percent` only; GB conversion is not available. Design-authored label was changed to match the emitted data. | Broadcaster enhancement to emit both fields, then swap label. |
| **KR-41.1-05** | **Telemetry slots stubbed.** `telemetry.slotsUsed = 0` and `telemetry.slotsMax = 8` are stubbed in the adapter because the backend does not expose Ollama slot counts on the snapshot. | Backend contract does not carry slot data; UI falls back to the design's default placeholder. | Ollama-slot instrumentation in ResourceGovernor + adapter update. |
| **KR-41.1-06** | **ReplayBar rewind control disabled.** The Rewind button in the design's replay bar is rendered disabled because the backend has no `/api/replay/rewind` endpoint. | Replay flow is forward-only today; backward scrubbing is out of scope. | New `/api/replay/rewind` endpoint + enabling the control. |
| **KR-41.1-07** | **CycleHistory columns show em-dash placeholders.** `history.jsx` renders em-dash placeholders for `consensus`, `flips`, `duration`, and `shocks` because `/api/replay/cycles` does not include them. | Backend cycle-list payload is a minimal summary today. Column headers remain so the table shape matches the design. | Extend `/api/replay/cycles` response shape + adapter + populate columns. |
| **KR-41.1-08** | **ErrorState event log is illustrative.** `states.jsx` event-log lines are the design's copy rather than live events, because the backend emits no structured error stream yet. | Backend has no error-event channel; the surface displays the design's static vocabulary so users still see a meaningful empty state. | Backend structured error stream + adapter + live rendering. |
| **KR-41.1-09** | **"Top Influencers" ranking uses `agent.confidence` DESC.** Ranking is by `StateFrame.agents[].confidence` descending — **not** by CONTRACT §2.7 `Report.influence.outDegree`, which is a post-cycle report field not emitted during live view. | User-authorized sort choice (pre-authorized decision point (i)). `confidence` is already on the live frame; no backend change needed for the v1 React port. | Backend emit of live `influence` + swap sort key. |
| **KR-41.1-10** | **Rationale citations / sources stubbed `[]`.** The rationale/advisory adapter stubs `citations: []` and `sources: []`; `RationaleFeed` renders "—" where citations would live. Neither the WebSocket `Rationale` stream nor `/api/advisory/*` populate these arrays in this phase. | User-authorized stub (pre-authorized decision point (ii)). Keeps types stable. VALIDATION row 13 tracks this. | Backend enhancement to emit citations + populated React rendering in a future phase. |
| **KR-41.1-11** | **Interview modal is non-streaming.** `modals.jsx` → `/api/interview/{agent_id}` issues a single POST and renders the full response body when it returns. Token-by-token SSE/WS streaming is **not** implemented. | User-authorized (pre-authorized decision point (iii)). Matches current REST shape; the design export ships no SSE parser. VALIDATION row 18 tracks this. | Backend SSE/WS + React stream consumer in a dedicated phase. |

**UAT enforcement rule:** A VALIDATION row whose expected behavior is marked as tracked by a KR passes if the observed behavior matches the KR's Scope. UAT failure requires that the deviation fall outside every KR listed here. New deviations discovered during UAT MUST be filed as new KR-IDs (Phase 41.1 addendum) or as bugs — they are not silently tolerated.
