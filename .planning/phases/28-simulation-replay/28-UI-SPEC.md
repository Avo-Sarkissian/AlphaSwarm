# Phase 28 -- UI Design Contract

> Visual and interaction contract for Simulation Replay. Generated by gsd-ui-researcher. Adapted for Textual (Python terminal UI) -- not a web application.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | Textual (>=8.1.1) |
| Preset | Custom dark theme `"alphaswarm"` via `textual.theme.Theme` (established Phase 9) |
| Component library | Textual built-in widgets + existing custom widgets (AgentCell, HeaderBar, RationaleSidebar, TelemetryFooter, BracketPanel) |
| Icon library | Unicode symbols only (no icon library) |
| Font | Terminal monospace (user's terminal font, no override) |

**Note:** This is a Python terminal UI. shadcn, Tailwind, and browser-based design systems do not apply. All styling uses Textual CSS (TCSS in `DEFAULT_CSS` strings) and the Textual theme variable system. Phase 28 adds no new CSS files -- all styling is inline `DEFAULT_CSS` per established pattern.

---

## Spacing Scale

Terminal UIs operate in character cells (1 unit = 1 character width or 1 line height). All spacing values are in cell units, not pixels. Inherited from Phase 9 UI-SPEC without modification.

| Token | Value | Usage |
|-------|-------|-------|
| none | 0 | Cell-to-cell adjacency in agent grid |
| xs | 1 | Inline padding within header labels, cell internal margin |
| sm | 2 | Gap between header sections (pipe-separated) |
| md | 3 | Padding around the grid container |
| lg | 4 | Margin between header bar and grid |

Exceptions: None for Phase 28. Agent grid cells remain exactly 3 characters wide by 1 line tall.

---

## Typography

Terminal UI is monospace. "Size" refers to visual emphasis via weight and style, not pixel size. Inherited from Phase 9 UI-SPEC.

| Role | Style | Weight | Usage |
|------|-------|--------|-------|
| Body | normal | normal (400) | Agent cell content (color-only), rationale sidebar text, telemetry values |
| Label | normal | normal (400) | Header status text, round counter, elapsed time, replay round indicator |
| Heading | normal | bold (700) | App title "AlphaSwarm" in header, sidebar/panel section titles |
| Status indicator | normal | bold (700) | Active phase label, **replay badge "REPLAY -- Cycle {id}"** |

Only 2 weights used: **normal** (400) for data and **bold** (700) for emphasis. No italic, no underline.

|------|------------|-------|-------|

## Component Inventory

### Existing Components (Reused Without Modification)

These widgets are reused as-is during replay. `ReplayStore` writes snapshots that feed the same rendering pipeline.

| Component | Role in Replay |
|-----------|---------------|
| `AgentCell` | Displays replayed agent signals via `update_color()`. Click handler is **disabled** during replay (interview requires live LLM). |
| `AgentGrid` (Container `#agent-grid`) | 10x10 grid, unchanged layout. |
| `RationaleSidebar` | Displays rationale entries loaded on-demand per replayed round via `add_entry()`. |
| `BracketPanel` | Displays bracket summaries loaded on-demand per replayed round via `update_summaries()`. |
| `TelemetryFooter` | Displays static/idle metrics during replay (no live governor data). Shows replay status instead. |

### Modified Components

#### HeaderBar (Extended)

The existing `HeaderBar._render_header()` method is extended (or a new `_render_replay_header()` is added) to display the replay badge.

**Live mode header format (unchanged):**
```
  AlphaSwarm  |  Round 1/3  |  * Seeding  |  00:01:23
```

**Replay mode header format (new):**
```
  AlphaSwarm  |  REPLAY -- Cycle a1b2c3  |  Round 2/3  |  [AUTO]
```

Header layout specification:
- Position 1: "AlphaSwarm" -- bold `#4FC3F7` (unchanged)
- Position 2: "REPLAY -- Cycle {short_id}" -- "REPLAY" in bold `#FFA726`, " -- Cycle " in `#78909C`, "{short_id}" in `#E0E0E0`. The `short_id` is the first 8 characters of the cycle UUID.
- Position 3: "Round X/3" -- `#E0E0E0` (unchanged format)
- Position 4: Mode indicator -- "[AUTO]" in `#66BB6A` or "[PAUSED]" in `#78909C`
- Separator: ` | ` in `#78909C` (unchanged)

#### TelemetryFooter (Extended for Replay)

During replay, governor metrics (RAM, TPS, Queue, Slots) are not available. The footer displays replay-specific information instead.

**Replay footer format:**
```
  Mode: Replay  |  Cycle: a1b2c3d4  |  Round: 2/3  |  Space/Right: Next  P: Play/Pause  Esc: Exit
```

- All labels in `#78909C` (secondary)
- All values in `#E0E0E0` (foreground)
- Key hints in `#4FC3F7` (primary) for discoverability

### New Components

#### CyclePickerScreen (Screen overlay -- Claude's discretion)

A simple selection overlay shown when multiple completed cycles exist. Follows the `RumorInputScreen` / `InterviewScreen` overlay pattern.

**Layout:**
```
+--------------------------------------------+
|         Select Simulation Cycle            |
|                                            |
|  > 2026-04-12 14:30 — a1b2c3d4  (latest)  |
|    2026-04-11 09:15 — e5f6g7h8             |
|    2026-04-10 16:45 — i9j0k1l2             |
|                                            |
|  Up/Down to select  ·  Enter to replay     |
|  Esc to cancel                             |
+--------------------------------------------+
```

**Styling:**
- Screen background: `#121212` (`$background`)
- Container border: `solid #4FC3F7` (`$primary`) -- matches `RumorInputScreen` pattern
- Container background: `#1E1E1E` (`$surface`)
- Container width: 56 characters
- Title: "Select Simulation Cycle" in bold `#4FC3F7`
- Selected row: background `#1E1E1E` with text in `#4FC3F7` (highlighted)
- Unselected rows: text in `#E0E0E0`
- "(latest)" suffix: `#78909C` (secondary, muted)
- Hint text: `#78909C`

**Behavior:**
- Up/Down arrow keys navigate the list
- Enter selects and dismisses the screen with the chosen cycle_id
- Escape cancels and returns to COMPLETE state
- If only one cycle exists, skip this screen entirely and auto-select it
- Maximum 10 most recent cycles displayed (scrollable if more exist, but practically unlikely)

## Interaction Contract

### Key Bindings

#### AlphaSwarmApp Level (Global)

| Key | Action | Gate Condition | Description |
|-----|--------|---------------|-------------|
| `r` | `action_start_replay` | `SimulationPhase.COMPLETE` | Enter replay mode. Opens CyclePickerScreen if multiple cycles, else auto-selects latest. |
| `q` | `action_quit` | Always | Quit application (existing binding, unchanged) |
| `s` | `action_save_results` | `SimulationPhase.COMPLETE` | Save results (existing binding, unchanged). **Disabled during REPLAY** -- gate on `phase == COMPLETE`. |

#### Replay Mode Bindings (Active Only During Replay)

These bindings are active only when `SimulationPhase` is `REPLAY` (or a replay flag is True on AppState).

| Key | Action | Description |
|-----|--------|-------------|
| `p` | `action_toggle_replay_mode` | Toggle between auto-advance and manual step mode |
| `space` | `action_replay_next_round` | Advance to next round (manual mode only; ignored in auto mode) |
| `right` | `action_replay_next_round` | Same as Space -- advance to next round in manual mode |
| `escape` | `action_exit_replay` | Exit replay mode, restore COMPLETE state with original StateStore snapshot |

#### CyclePickerScreen Bindings

| Key | Action | Description |
|-----|--------|-------------|
| `up` | Navigate selection up | Move highlight to previous cycle |
| `down` | Navigate selection down | Move highlight to next cycle |
| `enter` | Select and dismiss | Dismiss screen with selected cycle_id |
| `escape` | Cancel | Return to COMPLETE state without starting replay |

#### Disabled During Replay

These interactions are suppressed when replay is active:

| Interaction | Normal Behavior | Replay Behavior | How Suppressed |
|-------------|----------------|-----------------|----------------|
| `AgentCell.on_click` | Opens interview | No-op with notification | Gate on `phase != REPLAY` (extend existing `COMPLETE` check) |
| Shock injection controls | Opens shock input | N/A (already gated on simulation phase) | `REPLAY` is not a round phase, so shock window never opens |
| `s` (save results) | Saves to markdown | No-op with notification | Gate on `phase == COMPLETE` (not `REPLAY`) |
| `r` (start replay) | Enters replay | No-op (already in replay) | Gate on `phase == COMPLETE` |

### Replay Flow

#### Entry Flow

1. User presses `r` when `SimulationPhase.COMPLETE`
2. If multiple completed cycles exist in Neo4j: push `CyclePickerScreen`
3. If only one cycle (or cycle selected): begin replay
4. Set `SimulationPhase.REPLAY` (or set replay flag on AppState)
5. Execute upfront query: `read_full_cycle_signals(cycle_id)` loads all 3 rounds of signal data
6. Header updates to replay format
7. Footer updates to replay format with key hints
8. Grid resets to all PENDING (`#333333`) cells
9. Auto-advance timer starts (3s interval via `set_interval(3.0, ...)`)
10. Round 1 data populates grid, sidebar, bracket panel

#### Round Transition Flow (Auto-Advance)

1. Timer fires after 3 seconds
2. Grid resets to PENDING for the new round (same visual reset as live simulation)
3. New round's agent signals populate cells from pre-loaded data
4. On-demand: `read_bracket_narratives()` and rationale entries load for the new round
5. Sidebar and bracket panel update
6. Header round counter increments

#### Round Transition Flow (Manual Step)

1. User presses `space` or `right`
2. Same round transition logic as auto-advance
3. No timer -- waits for next user input
4. "[PAUSED]" shows in header position 4

#### Exit Flow

1. User presses `escape` during replay
2. Auto-advance timer is cancelled
3. `SimulationPhase` returns to `COMPLETE`
4. Original `StateStore` snapshot from the completed simulation is restored
5. Header reverts to live "Complete" format
6. Footer reverts to live telemetry (or idle if no governor metrics cached)
7. Grid shows final Round 3 state from the original simulation
8. All replay key bindings become inactive; `r` binding becomes active again

#### End-of-Replay Flow

1. After Round 3 is displayed and 3s timer fires (or user advances past Round 3)
2. Grid remains showing Round 3 replay data (no reset)
3. Header shows "Round 3/3" with "[DONE]" replacing mode indicator -- "[DONE]" in `#78909C`
4. User can press `escape` to exit replay, or `r` is re-enabled to start another replay
5. No auto-loop, no auto-exit

### Notification Messages

| Trigger | Message | Severity | Timeout |
|---------|---------|----------|---------|
| Replay started | "Replaying cycle {short_id} -- 3 rounds, 3s per round" | information | 5s |
| Manual mode activated | "Manual mode -- Space or Right to advance" | information | 3s |
| Auto mode activated | "Auto-advance mode -- 3s per round" | information | 3s |
| Replay complete (Round 3 shown) | "Replay complete -- Esc to exit" | information | 0 (persistent) |
| Agent click during replay | "Interviews unavailable during replay" | warning | 3s |
| Save during replay | "Save unavailable during replay" | warning | 3s |
| No completed cycles found | "No completed simulation cycles found" | error | 5s |
| Neo4j not connected | "Neo4j not connected -- cannot replay" | error | 5s |
| Cycle data load failure | "Failed to load cycle data: {error}" | error | 0 (persistent) |

## State Mapping

### SimulationPhase Extension

Add `REPLAY` to the `SimulationPhase` enum in `types.py`:

| Enum Value | Display Label | Description |
|------------|--------------|-------------|
| `REPLAY` | "Replay" | TUI is replaying a past cycle from Neo4j data |

This is used as the gate condition for:
- Disabling agent click (interview)
- Disabling save results
- Disabling shock controls
- Enabling replay-specific key bindings
- Switching header/footer to replay format

### ReplayStore

A new class (separate from `StateStore` per prior decision) that feeds the same `StateSnapshot`-shaped data to the TUI rendering pipeline.

**Key properties:**
- Holds pre-loaded signal data for all 3 rounds (from `read_full_cycle_signals()`)
- Exposes a `snapshot()` method matching `StateStore.snapshot()` return type
- Phase is always `SimulationPhase.REPLAY`
- Governor metrics are `None` (no live governor during replay)
- TPS is `0.0` (no live inference)
- Rationale entries and bracket summaries are populated on-demand per round

### AppState Integration

The `AlphaSwarmApp` instance needs to know which store to poll during `_poll_snapshot()`:
- During live simulation: poll `self.app_state.state_store.snapshot()`
- During replay: poll `self._replay_store.snapshot()` (where `_replay_store` is a `ReplayStore` instance)

This is an internal implementation detail. The TUI rendering code (`_poll_snapshot`) remains unchanged -- it just receives snapshots from a different source.

## Accessibility Notes

- Replay mode badge uses `$warning` (`#FFA726`) which has sufficient contrast (>4.5:1) against `$surface` (`#1E1E1E`)
- "[PAUSED]" and "[AUTO]" mode indicators use distinct colors (`#78909C` and `#66BB6A`) that are distinguishable even in reduced color vision
- Key binding hints in the footer provide text-based discoverability for all replay controls
- Notification messages describe all state transitions in text, not just color
- The 3-second auto-advance delay is slow enough for screen readers to announce round transitions
