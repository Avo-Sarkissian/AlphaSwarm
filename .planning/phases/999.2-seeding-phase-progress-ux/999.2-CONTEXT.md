# Phase 999.2: Seeding Phase Progress UX - Context

**Gathered:** 2026-05-03
**Status:** Ready for planning

<domain>
## Phase Boundary

Add live progress indicators during the agent seeding phase so users know what's happening before ROUND_1 starts.

Currently, `SeedingState` shows a frozen `000/100` counter for the entire duration of `inject_seed()` (30–120s) because `agent_states` is empty until ROUND_1 agents actually vote. The overlay then exits before the 0→100 agent count climb ever becomes visible.

This phase closes the gap with **frontend-only** changes to `SeedingState`, `app_v2.jsx`, and `TaskBanner.tsx`. No backend schema changes.

</domain>

<decisions>
## Implementation Decisions

### inject_seed feedback (SEEDING phase, agents.length === 0)
- **D-01:** While `phase === 'seeding'` and no agents exist, replace the frozen `000/100` display with: an elapsed timer (seconds since phase flipped to seeding) and an Ollama model health chip (`ORCHESTRATOR LOADED` / `LOADING MODEL…`).
- **D-02:** Elapsed timer reads `snap.elapsed_seconds` from the existing WebSocket snapshot (already emitted by `StateStore.snapshot()` via `time.monotonic() - _start_time`). No new backend field needed.
- **D-03:** Ollama health chip reuses the existing `useOllamaHealth` hook (already imported in `TaskBanner.tsx`). Mount or call it conditionally in `SeedingState` only when `phase === 'seeding'`.

### Agent count wiring (SEEDING → ROUND_1 transition)
- **D-04:** Extend the `SeedingState` overlay into early ROUND_1: show the overlay when `phase === 'seeding' || (phase === 'round_1' && agents.length < 100)`. During this window, `agents.length` climbs from 0 → 100 using real `agent_states` data — no fake counter.
- **D-05:** Once `agents.length === 100` (all agents have voted), the overlay yields to the main dashboard. The force graph and bracket panel take over. This is the natural exit condition with no extra state needed.

### TaskBanner wording
- **D-06:** During `phase === 'seeding'`, `TaskBanner` displays `SEEDING · mm:ss` instead of `SIMULATION RUNNING`. Once phase flips to `round_1`, banner updates to `SIMULATION RUNNING · mm:ss` as before.
- **D-07:** The `running` prop currently gates the TaskBanner. A new `phase` prop (or derived `isSeeding` boolean) must be threaded through `app_v2.jsx` → `TaskBanner` to enable the label switch. Keep the Cancel button present during seeding (cancellation is always allowed).

### Backend changes
- **D-08:** Frontend-only. Zero changes to `state.py` (`StateSnapshot`), `simulation.py`, or any backend route. All data needed (elapsed_seconds, agent_states, Ollama health) is already available.

### Claude's Discretion
- Exact copy / label text for Ollama health chip inside SeedingState (`ORCHESTRATOR LOADED` vs `OLLAMA READY` vs similar).
- Whether `useOllamaHealth` is called directly in `SeedingState` or the health result is passed down as a prop from `app_v2.jsx`.
- Progress bar behavior during SEEDING phase (pulse animation vs stay at 0 until R1 starts).
- CSS/styling tweaks to accommodate the timer + health chip in the seeding-center layout.

</decisions>

<specifics>
## Specific Ideas

- The seeding-center layout in `states.jsx` already handles the agent ID stream and bar. For the SEEDING sub-window (agents.length === 0), the top kicker should switch from `SEEDING · 000/100` to the elapsed timer + health chip. Once agents.length > 0, the existing `SEEDING · NNN/100` counter resumes — same component, two branches on `n === 0`.
- The `useOllamaHealth` hook accepts an `enabled` flag to gate polling. Pass `phase === 'seeding'` so polling stops once seeding ends.

</specifics>
