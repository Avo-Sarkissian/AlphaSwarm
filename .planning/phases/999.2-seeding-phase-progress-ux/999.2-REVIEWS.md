# Cross-AI Plan Review — Phase 999.2: Seeding Phase Progress UX

## Gemini Review

### 1. Summary
The implementation plan for Phase 999.2 is well-structured and focuses on critical UX improvements for the simulation's initialization phase. It correctly identifies and removes the `demoState` bottleneck in `app_v2.jsx`, which previously blocked real-time state transitions. The approach is surgical, leveraging existing hooks (`useOllamaHealth`, `useTelemetry`) to maintain consistency and minimize backend impact. The two-wave structure provides a clear separation between component logic updates and high-level routing changes.

### 2. Strengths
- **Strategic De-blocking:** Removing `demoState` is a crucial "finding" that ensures the real-time simulation state actually reaches the UI.
- **Defensive Formatting:** The use of `Number(seconds) || 0` in `fmtSeedingElapsed` demonstrates good defensive programming against potential `null` or `undefined` values from early WebSocket packets.
- **Resource Efficiency:** Conditionally enabling the `useOllamaHealth` polling only during the seeding phase prevents unnecessary background requests once the simulation is in full swing.
- **Extended Overlay Logic:** The decision to keep the `SeedingState` overlay active until `agents.length === 100` in Round 1 (D-04) elegantly handles the transition period where agents are still populating the graph.

### 3. Concerns

- **TaskBanner Timer Omission (MEDIUM):**
  D-06 asks for `SEEDING · mm:ss`. Plan 01 Task 2 adds `phase === 'seeding' ? 'SEEDING'` as the taskName branch. Whether this produces `SEEDING · mm:ss` in the rendered output depends on TaskBanner's existing timer rendering — worth confirming the `fmtElapsed` wrapper is also applied to the seeding label path.

- **React Hook Rule Concern (MEDIUM):**
  Plan 01 Task 1 notes "Mount/call it conditionally in SeedingState only when phase === 'seeding'". The plan's actual action already follows the correct unconditional pattern (`useOllamaHealth(phase === 'seeding')`) — flagged as a clarity concern, not a real violation in the written plan.

- **Prop-Drilling for Telemetry in TaskBanner (LOW):**
  If TaskBanner renders `SEEDING · mm:ss`, it needs elapsed time access. The plan should confirm this is already wired via an existing source rather than requiring a new prop.

- **CSS Class Dependencies (LOW):**
  Task 1 renders into `.state-kicker`. Verify these classes exist from Phase 41.1 before executing.

### 4. Suggestions

- Confirm that `taskName = 'SEEDING'` in TaskBanner produces `SEEDING · mm:ss` in the rendered output (via existing `fmtElapsed` wrapper) — add explicit note to the executor.
- Verify `<TaskBanner>` is mounted outside `canvas-wrap IIFE` so it remains visible during the overlay.
- Consider moving `fmtSeedingElapsed` to a shared utility if both components will need the same `mm:ss` logic.

### 5. Risk Assessment

**Overall Risk: LOW**

Frontend-only change with no backend modifications. Most risks are surfaced by `npm run typecheck` and the human-verify checkpoint.

## Codex Review

### 1. Summary
The plans are directionally solid: they stay frontend-only, isolate the work into sensible waves, and correctly identify the real blocker — the `demoState` sentinel in `app_v2.jsx`. The main weaknesses are architectural coherence and a few implementation-detail mismatches: Wave 2 proposes a second overlay-routing table instead of extending the existing one in `states.jsx`, the timer field name in the plan does not match the frontend type shape, and the verify commands do not match the repo scripts.

### 2. Strengths
- Scope is appropriately minimal: plans stay inside `states.jsx`, `TaskBanner.tsx`, and `app_v2.jsx`, which aligns with D-08.
- Wave split is sensible — Wave 1 makes components capable, Wave 2 wires the live render path.
- `useOllamaHealth(phase === 'seeding')` is the correct React pattern: avoids conditional hook calls while still stopping polling when seeding ends.
- Plans reuse existing telemetry and health plumbing rather than inventing new snapshot fields.
- Manual live checkpoint is justified — build/typecheck can catch prop wiring, but not the seeding-to-round transition timing.
- Threat modeling is proportionate — no new user-input surfaces or HTML injection.

### 3. Concerns

- **HIGH: Split overlay routing path.** Wave 2 duplicates overlay routing in `app_v2.jsx` instead of extending the existing precedence table in `LifecycleOverlay` (in `states.jsx:225`). This creates a split-brain UI path where `reconnectFailed` and memory-pause precedence can be lost during seeding or early R1, and leaves `MemoryPausedState`/`ErrorState` still un-restored in the live canvas path.

- **MEDIUM: Field name mismatch — `snap.elapsed_seconds` vs `telemetry.elapsedSeconds`.** The plan text says to read `snap.elapsed_seconds`, but the React layer receives `telemetry.elapsedSeconds` after adaptation in `types.ts`. Implemented literally, this is a real wiring bug, not just wording drift.

- **MEDIUM: Wrong verification command.** `frontend/package.json` exposes `npm run check` and `npm run build`. There is no `npm run typecheck` and no root `package.json`. The acceptance criteria checks will fail as written.

- **MEDIUM: Chip label matrix is underspecified.** `useOllamaHealth` and `health.py` allow `connected=true, models_loaded=[]` and `connected=false, models_loaded=[]`. A pure count-based mapping can confuse "loading" with "offline."

- **LOW: Timer drift.** `TaskBanner` currently uses a local wall-clock timer while `SeedingState` is supposed to use WebSocket telemetry. If parity matters, the two timers can drift after reconnects or remounts.

- **LOW: Optional `phase?` hides missed wiring.** Once Wave 2 lands there is only one caller — keeping it optional makes missed wiring easier to hide.

- **LOW: Pre-agent branch may fire in wrong phase.** The pre-agent branch is keyed to `n === 0`. If `phase` flips to `round_1` before the first populated agent frame arrives, it may still show the pre-agent timer/chip path unless also gated by `phase === 'seeding'`.

### 4. Suggestions

- Rework Wave 2 to route through the existing `LifecycleOverlay` logic (or explicitly copy its precedence: `reconnectFailed` → `idle` → seeding/early-R1 → memory pause).
- In frontend code, use `telemetry.elapsedSeconds` — not `snap.elapsed_seconds` (backend field name).
- Fix verification steps to `cd frontend && npm run check && npm run build`.
- Define the Ollama chip state matrix explicitly: `disconnected`, `connected with no models`, `connected with models loaded`. Document intentional collapses.
- Add a small manual UAT matrix: cold start, warm start, early-R1 `0→100` climb, forced Ollama disconnect, cancel during seeding.
- After Wave 2, make `phase` required in `TaskBannerProps` unless a second caller appears.
- Tighten the SeedingState branch condition to match D-01 exactly: `phase === 'seeding' && agents.length === 0`.

### 5. Risk Assessment

**Overall risk: MEDIUM.** The plans should achieve the visible UX goal if implemented carefully, but routing and data-shape ambiguity could produce edge-case regressions. Tightening overlay precedence, timer field naming, and verification commands reduces risk substantially.
