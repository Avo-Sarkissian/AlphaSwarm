# Phase 41.4 — Model Overhaul Decision Log

## Final decision (2026-05-06)

**Worker:** `qwen3:8b` (Q4_K_M, ~5 GB)
**Orchestrator:** `qwen3.6:27b-q4_K_M` (~17 GB)
**Orchestrator thinking mode:** **OFF** (`think: false` in API calls)
**Ollama env:** `OLLAMA_NUM_PARALLEL=4`, `OLLAMA_MAX_LOADED_MODELS=2`, `OLLAMA_KEEP_ALIVE=5m`

Estimated full sim time: **~6 min** (vs ~26 min on prior qwen3.5 stack — ~5× speedup).

---

## Why this combination

### Worker = qwen3:8b
- `qwen3moe` architecture supports parallel requests in ollama 0.21.2 (confirmed empirically: 8 concurrent calls completed in 7.5s wall, 100% GPU, 14 tok/s/slot).
- Replaces `qwen3.5:9b` which was forced to `Parallel=1` by the `qwen35` architecture limitation (ollama issues #14510, #14621, #14879 — open, no upstream fix).
- ~74 MMLU vs qwen3.5:9b's ~76 — quality drop is negligible for persona-driven JSON output.

### Orchestrator = qwen3.6:27b
- 27b base model with stronger reasoning than qwen3.5:35b at smaller size.
- Architecture is `qwen35moe` so parallel is forced to 1 — irrelevant because orchestrator runs 1–2 sequential calls per cycle anyway.
- Q4_K_M tag chosen specifically to avoid the IQ4_XS Metal-kernel regression (~3.8× slowdown on Apple Silicon).

### Thinking = OFF
Empirical probe results (2026-05-06):
| Mode | Wall per call | Output | Thinking generated |
|---|---|---|---|
| `think: true` | **274.6s** | 52 tok | 5760 chars (~1500 tok) |
| `think: false` | 9.5s | 53 tok | empty |

- Thinking adds **~26 min per sim** (6 orch calls × 265s extra each).
- Both outputs produced valid JSON with all required schema keys.
- Thinking output was *slightly* more quant-structured; non-thinking was more grounded ("unverified rumor"). No clear quality winner for the consensus-aggregation workload.
- Conclusion: 26 minutes per sim is too steep for a marginal, non-obvious quality gain.

---

## How to revisit / flip thinking back ON

If future experiments suggest the orchestrator is making weaker consensus decisions and you want deeper reasoning:

1. **One-shot toggle** (test before committing): pass `think=True` in the orchestrator's `client.chat()` call site. No model reload needed.
2. **Permanent flip**: update the `think=` argument in the orchestrator chat call (search codebase for the orchestrator client invocation; likely in `src/alphaswarm/orchestrator.py` or similar).
3. **Hybrid pattern**: keep `think=False` as default, but pass `think=True` only on specific high-stakes calls (e.g., final-round consensus, divergence_flag follow-ups).

No model swap, no Modelfile edit, no `ollama create` rerun — qwen3.6:27b supports both modes natively.

### Decision triggers to revisit

- Consensus quality complaints (orchestrator missing nuance in agent output)
- Divergence-flag false negatives (failing to flag obvious disagreement)
- Wanting to A/B compare same sim with thinking on vs off

## Empirical data (this session)

### Worker probe — qwen3:8b @ NUM_PARALLEL=4
- 8 concurrent calls, NUM_PARALLEL=16 (initial bug): **45.55s wall**, model spilled to 50/50 CPU/GPU, 103 GB virtual KV allocation
- 8 concurrent calls, NUM_PARALLEL=4 (after fix): **7.53s wall**, 100% GPU, 29 GB allocation, ~14 tok/s/slot
- Verdict: **NUM_PARALLEL=16 is too high for this hardware**; KV cache bloat kicks model out of GPU. **N=4 is the sweet spot.**

### Orchestrator probe — qwen3.6:27b-q4_K_M
- `think=True`: 274.65s wall, 1500 thinking tokens + 52 output tokens, JSON parses cleanly
- `think=False`: 9.55s wall, 53 output tokens, JSON parses cleanly
- 100% GPU, 42 GB allocation (262k default context — could trim to 8k for orchestrator workload if RAM pressure returns)

### Confirmed bugs in current ollama 0.21.2
- `qwen35` / `qwen35moe` architecture forced to `Parallel=1` regardless of NUM_PARALLEL env var
- IQ4_XS quants have ~3.8× slowdown vs Q4_K_M on Apple Silicon (Metal kernel regression)
- Thinking-parameter regression in some post-2026.4.15 builds (NOT affecting our 0.21.2)
