## Bug A ‚Äî `/ws/state` upgrade dropped by Vite proxy (HIGH, ~5-line fix)

### Symptom
Browser console (during smoke): `WebSocket connection to 'ws://localhost:5173/ws/state' failed: WebSocket is closed before the connection is established`. UI never receives live state ‚Äî keeps rendering whatever frame it last had while client-side `ELAPSED` counter increments. Makes any backend slowness look like a complete hang.

### Root cause
`frontend/vite.config.ts:9-12`:
```ts
proxy: {
  '/api': { target: 'http://localhost:8000', changeOrigin: true },
  '/ws':  { target: 'ws://localhost:8000', ws: true },   // ‚Üê missing changeOrigin
}
```
The `/ws` entry has `ws: true` but no `changeOrigin: true`. Vite forwards the browser `Origin: http://localhost:5173` header upstream; some FastAPI/uvicorn WS upgrade paths reject it because it doesn't match the target host. Browser sees the upgrade fail and reports the localhost:5173 URL.

The frontend WS URL construction is correct ‚Äî `useWebSocket.ts:34-35` uses `window.location.host`, which during dev is `localhost:5173` and proxies via Vite. Fix is on the proxy side, not the hook.

### Fix
Add `changeOrigin: true` to the `/ws` entry. One-line change.

### Verification
- `git checkout` the 41.4 branch, `npm --prefix frontend run dev`, hard-refresh page.
- Expect: no WS error in console; `useWebSocket` `connected` flips true; live snapshots flow on every backend tick.

## Bug B ‚Äî R3 ollama generate latency 14‚Äì20 min per call (HIGH, investigation needed)

### Symptom
During the smoke run, ollama serve terminal showed every `POST /api/generate` returning `200` with durations of **14m / 15m / 20m** per single agent decision. With 100 agents in R3 this projects to days. GPU pinned at 100%, so inference was happening ‚Äî just per-call latency was wrong by 2‚Äì3 orders of magnitude vs. expected (M1 Max 64GB, quantized 7-13B model expected ~10-50 tok/s).

### Likely root causes (ranked by prior-art on this codebase)

1. **R3 prompt context bloat.** R3 prompts include peer summaries from R1+R2. If the builder concatenates raw decision text instead of summarized signals, total context can exceed the model's `num_ctx`. Ollama silently re-runs with a larger context window, ballooning latency. Code surface: simulation R3 prompt builder + worker invocation; check `src/alphaswarm/simulation.py` for the round=3 prompt assembly path and what gets passed into `worker.invoke()`.

2. **Model swap thrash.** `OllamaModelManager` (`src/alphaswarm/ollama_models.py:23-114`) does sequential load ‚Üí unload ‚Üí load between orchestrator and worker, with `keep_alive="5m"`. If the simulation pops the orchestrator between every worker call (instead of per-round), each transition is a full model reload. Verify by tailing ollama serve log for `loading model` / `unloading model` events vs. count of `/api/generate`.

3. **Worker Modelfile size mismatch.** `modelfiles/Modelfile.worker` was modified by Codex's pass ‚Äî confirm the `FROM` tag still points at a model that fits comfortably in 64GB unified memory alongside the orchestrator. A 70B q4_0 + 13B q4_0 will swap, a 13B + 7B will not.

4. **GPU layer offload misconfigured.** If `num_gpu` is unset on the worker Modelfile, ollama may split layers between GPU and CPU. With M1 Max unified memory there's no benefit to splitting ‚Äî should be all on GPU.

### Investigation steps (Wave 1 ‚Äî diagnose before fix)

- [ ] Capture a fresh ollama serve log during a controlled R1-only run (skip R2/R3) and compare durations.
- [ ] Add structlog instrumentation in `worker.invoke()` to log prompt token count and model tag per call.
- [ ] Run `ollama show alphaswarm-worker --modelfile` to confirm context size, parameters, base model.
- [ ] Tail ollama serve and count `loading model` events for one full 3-round cycle.

### Fix candidates (Wave 2 ‚Äî only after diagnosis confirms which root cause)

- If context bloat: trim R3 prompt to bracket-aggregated signal histogram + top-5 peer narrative summaries, cap total prompt at ~4-6k tokens.
- If swap thrash: load orchestrator once at sim start with `keep_alive="-1"` (forever) until simulation end; same for worker. Reorder simulation so all worker calls in a round complete before orchestrator is invoked.
- If Modelfile mismatch: pin to known-good model tags and document RAM budget.
- If GPU offload: explicit `PARAMETER num_gpu 999` in both Modelfiles.

## Bug C ‚Äî `/api/edges/{cycle_id}?round=N` returns 422 / 500 (LOW, cosmetic)

### Symptom
Browser network tab: 5 failures during smoke run. `round=0` ‚Üí 422 Unprocessable Entity. `round=1,2,3` ‚Üí 500 Internal Server Error.

### Diagnosis
- 422 is correct backend behavior: `src/alphaswarm/web/routes/edges.py:32` enforces `Query(..., ge=1, le=3)`. **Frontend is requesting `round=0` somewhere ‚Äî wrong.**
- 500s on valid rounds indicate `read_influence_edges` raises during a partial-write window. Likely: cycle exists but R1 INFLUENCED_BY edges aren't all written yet when polling first hits.

### Fix
- Frontend: change the `round=0` caller to `round=1` (or skip the call before R1 finishes). Grep target: `frontend/src` for `/api/edges` callers.
- Backend: wrap `read_influence_edges` in a try and return `EdgesResponse(edges=[])` for empty/in-flight reads instead of bubbling Neo4jError as 500.
