## Bug B — diagnosis result

Original 4 candidates from CONTEXT.md were:
1. R3 prompt context bloat
2. Model swap thrash
3. Worker Modelfile size mismatch
4. GPU layer offload misconfigured

**None of these were the actual root cause.**

**Actual root cause:** ollama 0.21.2 hardcodes `Parallel=1` for `qwen35` /
`qwen35moe` architecture. Server log line:

```
WARN msg="model architecture does not currently support parallel requests" architecture=qwen35
```

Empirical evidence captured this session:
- Single inference, realistic prompt (~355 input tok): **4.62s total, 25 tok/s eval rate**
  → model itself is healthy on M1 Max.
- 8 concurrent calls: total durations **5.16, 10.41, 16.6, 22.51, 28.22, 32.54, 38.52, 44.81 s**
  — pure serial stacking despite `OLLAMA_NUM_PARALLEL=16`.
- ollama 0.21.2 runner load request explicitly shows `Parallel:1`.
- Confirmed via GitHub issues #14510, #14621, #14879 — open, no fix from upstream.

The original 14-20 min/call CONTEXT.md report was **queue-time accumulating
on the later requests in the 100-agent wave**, not per-inference latency.
Per-inference is ~5s; the 100th queued call waits ~500s and ollama reports
that wait as part of `total_duration`.

**Important counterintuitive finding:** Qwen3.6 has the SAME parallel
limitation (also forced to `runner.parallel=1`). The "older" qwen3 / qwen3moe
architecture is the only Qwen family that currently parallelizes in ollama.
