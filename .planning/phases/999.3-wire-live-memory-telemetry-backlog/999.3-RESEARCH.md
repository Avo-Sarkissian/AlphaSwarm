# Phase 999.3: Wire live memory % telemetry — Research

**Researched:** 2026-05-03
**Domain:** Backend telemetry wiring (broadcaster + governor + StateStore + WebSocket frame)
**Confidence:** HIGH

## Summary

The KR-41.1-04 bug is a **wiring gap, not a measurement gap**. The repo already reads `psutil.virtual_memory().percent` in two places (`memory_monitor.py:106`, `web/routes/health.py:54`). The dashboard reads `governor_metrics.memory_percent` from the WebSocket snapshot via `frontend/src/adapter/frame.ts:56`. The KPI label `kpi-value` shows `0%` because two preconditions both fail simultaneously when nothing is going wrong:

1. **State-change gating.** `ResourceGovernor._apply_state_transition` only calls `_emit_metrics(reading)` when `old_state != new_state` (`governor.py:377-384`). In the steady-state RUNNING regime — which is most of every session — `_state_store.update_governor_metrics()` is never called, so `StateSnapshot.governor_metrics` stays at its `__init__` default of `None`.
2. **Lifecycle gating.** `governor.start_monitoring()` is invoked from `simulation.py:506,925` and torn down at `simulation.py:551,1218`. Outside an active simulation (idle dashboard, post-FINAL view, replay mode), the monitor task isn't even running, so no `MemoryReading` is ever produced and no metrics could be emitted even if the gating in (1) were removed.

The broadcaster, by contrast, runs whenever the FastAPI app is alive (`web/app.py:102` inside the lifespan, before `yield`). It already publishes a full `StateSnapshot` to all WS clients at 5Hz via `dataclasses.asdict(snap)`. The minimal fix is to make the broadcaster — not the governor — the source of truth for live memory %, since it is the surface that always runs at the 5Hz cadence the frontend expects.

**Primary recommendation:** Patch `snapshot_to_json` (or the `_broadcast_loop` tick) to compute `psutil.virtual_memory().percent` once per 200ms tick and overlay it onto the wire frame's `governor_metrics.memory_percent` field, falling back to the governor's own value when one is present (so an active simulation in CRISIS state still wins). No frontend change required — the adapter already reads `governor_metrics.memory_percent` and the KPI bar label already reads "%". Two-line change in `broadcaster.py`, optional one-line guard in `governor._apply_state_transition` to also emit on every tick (heartbeat) for symmetry with `tps`.

### Pattern 1: Broadcaster-side overlay (RECOMMENDED)

**What:** In `snapshot_to_json`, after `d = dataclasses.asdict(snap)`, mutate `d["governor_metrics"]["memory_percent"]` (creating the dict if `snap.governor_metrics is None`) using `psutil.virtual_memory().percent` whenever the governor hasn't supplied a fresher value.

**When to use:** This case — the broadcaster always runs, the governor only sometimes runs, and the wire frame is the single contract surface the frontend reads.

**Example (illustrative, planner refines):**

```python
# Source: derived from broadcaster.py:27-56 + state.py:17-29
# [VERIFIED: existing files in this repo]
import psutil
import time

def snapshot_to_json(state_store, replay_manager=None):
    if replay_manager is not None and replay_manager.is_active:
        snap = replay_manager.store.snapshot()
        return json.dumps(dataclasses.asdict(snap))

    snap = state_store.snapshot()
    rationales = state_store.drain_rationales(5)
    d = dataclasses.asdict(snap)
    d["rationale_entries"] = [dataclasses.asdict(r) for r in rationales]

    # MEM-01: live memory % overlay — always fresh, regardless of governor state.
    live_mem_pct = float(psutil.virtual_memory().percent)
    gm = d.get("governor_metrics")
    if gm is None:
        # IDLE / pre-simulation: synthesize a minimal metrics dict so the frontend
        # adapter's `gov.memory_percent` lookup hits a real number, not undefined.
        d["governor_metrics"] = {
            "current_slots": 0,
            "active_count": 0,
            "pressure_level": "green",
            "memory_percent": live_mem_pct,
            "governor_state": "idle",
            "timestamp": time.monotonic(),
        }
    else:
        # Active simulation: overlay only the memory_percent field; preserve
        # current_slots / governor_state / pressure_level which are governor-authoritative.
        gm["memory_percent"] = live_mem_pct
    return json.dumps(d)
```

### Pattern 2: Governor heartbeat (OPTIONAL, complementary)

**What:** In `_apply_state_transition`, call `self._emit_metrics(reading)` *unconditionally* on every monitor tick, not only when `old_state != new_state`. This makes `StateSnapshot.governor_metrics` non-stale during long-running RUNNING phases.

**When to use:** If/when downstream consumers need fresh `current_slots`, `active_count`, or `pressure_level` between transitions. The TUI footer (`tui.py:300-318`) would also benefit. Not strictly required for KR-41.1-04 if Pattern 1 ships, but harmonizes the metric path.

**Tradeoff:** Adds one StateStore write every 2.0s (default `check_interval_seconds`, `config.py:56`). StateStore writes are O(1) field assignment with no lock — negligible cost.

### Anti-Patterns to Avoid

- **Don't poll psutil on the React side.** The frontend has no Python; this would require a new REST endpoint and break the 5Hz unified-snapshot contract. WebSocket is the only delivery channel for live state per CLAUDE.md Hard Constraint 4.
- **Don't move `start_monitoring` into the lifespan as a workaround.** The governor's monitor task carries simulation-scoped state (crisis timer, debt tracking, slot adjustments) that is meaningless when no simulation is running. It also raises `GovernorCrisisError` when the monitor dies (`governor.py:217`), which would crash the broadcaster path. Keep the governor scoped to simulation runs.
- **Don't add `await asyncio.to_thread(psutil.virtual_memory)`** — psutil's macOS implementation is a Mach `host_statistics64` call returning in microseconds. The `health` route already calls `psutil.virtual_memory().percent` directly in an async handler (`web/routes/health.py:54`) — same pattern, same load profile.
- **Don't broadcast at >5Hz to "improve smoothness."** CLAUDE.md Hard Constraint 4 fixes the cadence at ~5Hz with drop-oldest backpressure. The fix must respect the existing tick.

## Common Pitfalls

### Pitfall 1: Replay mode shows host RAM, not replayed-cycle RAM
**What goes wrong:** `snapshot_to_json` has a fast-path for `replay_manager.is_active` (`broadcaster.py:45-47`) that returns early. If you add the psutil overlay in the live-mode branch only, replay mode still shows whatever `replay_manager.store.snapshot()` carried (currently `governor_metrics=None`, see `state.ReplayStore.snapshot()` at `state.py:284-294`).
**Why it happens:** Replay snapshots represent a *past* simulation; injecting host RAM into them is misleading.
**How to avoid:** Decide explicitly during discuss-phase. Two reasonable options:
1. (Recommended) Inject host RAM into replay frames too — the user is still on a live machine and the KPI label means "current host", not "host at replay-time". Lower surprise.
2. Show a sentinel (e.g., `null` → adapter falls back to `0`) and have the frontend hide the "Memory" KPI in replay mode.
**Warning signs:** Replay-mode UAT shows `Memory 0%` while live mode shows `Memory 67%` on the same hardware.

### Pitfall 2: psutil cached read returns stale percent under macOS App Nap
**What goes wrong:** psutil caches the last `virtual_memory()` reading in some configurations; on macOS, sleep/wake cycles can deliver the cached value for one tick.
**Why it happens:** psutil internal caching for sub-tick rate-limiting. In practice, virtual_memory() is NOT cached — it always issues a fresh `host_statistics64` call.
**How to avoid:** No action needed — `psutil.virtual_memory()` re-reads on each call. The `/health` endpoint relies on this behavior already with no observed staleness. `[VERIFIED: psutil source — virtual_memory has no @memoize decorator]`.
**Warning signs:** Memory KPI sticks at one value for >2s while system memory is visibly changing in Activity Monitor.

### Pitfall 3: `governor_metrics: None` survives the asdict round-trip and the adapter's null-coalesce eats it
**What goes wrong:** `dataclasses.asdict(snap)` on a snapshot where `governor_metrics is None` produces `{"governor_metrics": None}`. The adapter at `frame.ts:55` does `(r.governor_metrics ?? {})` then `gov.memory_percent` is `undefined` → falls back to `0`. The KPI shows 0% even after your patch ships if you only mutate the dict when `gm is not None`.
**Why it happens:** Snapshot defaults `governor_metrics=None` (`state.py:81`). Pre-simulation, post-simulation, and during replay, this is the live value.
**How to avoid:** In `snapshot_to_json`, synthesize a minimal `governor_metrics` dict when `d["governor_metrics"] is None` (see § Pattern 1 example). Always populate `memory_percent`; populate other fields with safe defaults (`current_slots: 0`, `governor_state: "idle"`, `pressure_level: "green"`).
**Warning signs:** Live frame shows `memory_percent` correctly but a frame captured during IDLE phase shows 0.

### Pitfall 4: TestClient WebSocket tests assert old contract
**What goes wrong:** `tests/test_web.py:345-361` asserts `"governor_metrics" in parsed` but does not check the field's value or shape. After the fix, IDLE-phase snapshots produce a non-None `governor_metrics`, which existing tests don't assert against — they pass — but new tests must lock the new contract or it will silently regress.
**How to avoid:** Add an explicit assertion: serialize a fresh `StateStore`, parse, and check `parsed["governor_metrics"]["memory_percent"] > 0`.
**Warning signs:** A code change reverts the synthesized-dict behavior and no test catches it.

### Pitfall 5: Governor heartbeat (Pattern 2) double-emits during state transitions
**What goes wrong:** If you implement Pattern 2 by adding `self._emit_metrics(reading)` *outside* the `if old_state != new_state` block, transitions emit twice (once unconditionally, once inside the conditional).
**How to avoid:** Replace the conditional emit, not augment it: move `self._emit_metrics(reading)` to fire on every tick *and* delete it from the transition log block. Keep the transition `log.warning(...)` line on its own.
